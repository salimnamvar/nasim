### C4-05: Architecture Pattern (Manager ‚Üí Adapter ‚Üí Store)

**Challenge:** Every Manager must have exactly one Adapter. No Manager ‚Üí Store direct calls.

**Action:** Inspected Component diagrams for Manager-Adapter-Store compliance.

**Result:**
| Group | Manager | Adapter | Store | Compliant? |
|-------|---------|---------|-------|------------|
| Agent | AgentOrchestrator | (none ‚Äî orchestrator calls other managers) | ‚Äî | N/A (orchestrating manager) |
| Safety | SafetyCoordinator | PermissionGate | (none ‚Äî pure logic) | PASS (no store needed) |
| Tool | ToolRegistry | (none ‚Äî registry IS the adapter) | Host FS | DEVIATION |
| Session | ‚Äî | SessionStore | Disk | DEVIATION |
| Config | ConfigLoader | (none) | File | DEVIATION |

**Issue:** The Tool Group uses ToolRegistry as both registry and adapter ‚Äî it directly accesses the Host Filesystem. The Config Group uses ConfigLoader without a distinct Adapter. These are **architectural deviations** from the strict Manager‚ÜíAdapter‚ÜíStore pattern. For Repository-layer groups (Tools, Session, Config), the "Store" IS the component ‚Äî the adapter pattern is collapsed. This is acceptable if documented but currently isn't explicitly justified in the C4 diagrams.

**Severity:** MEDIUM ‚Äî Architectural deviation from the mandated pattern. Should be documented as a design decision.
