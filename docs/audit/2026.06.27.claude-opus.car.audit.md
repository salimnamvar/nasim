## How to read this audit

Each finding is written in **CAR (Challenge тЖТ Action тЖТ Result)**:

- **Challenge** тАФ the rule/standard that applies and *why it matters*.
- **Action** тАФ exactly what was checked, the tool/command, and the evidence found.
- **Result** тАФ the verdict, the affected artifacts, and the precise fix.

Severity:

| Tag | Meaning |
| --- | --- |
| ЁЯФ┤ **CRITICAL** | Correctness defect or chain break that blocks merge (per `cicd.md`). |
| ЁЯЯа **HIGH** | Rule violation; must fix before sign-off. |
| ЁЯЯб **MEDIUM** | Should fix; degrades the design contract. |
| ЁЯФ╡ **LOW** | Polish / nuance / linter limitation. |
| ЁЯЯв **PASS** | Conforms тАФ recorded as a strength to preserve. |

# Part I тАФ C4 (Context / Container / Component)

## ЁЯЯа CAR-C4-01 тАФ Components are not inside a `Container_Boundary` (178├Ч, all 23 component diagrams)

**Challenge.** `c4.md` C4-C3-001 and the C4 model itself require a component
diagram to show *the internals of exactly one container*; components must sit inside
a `Container_Boundary`. The recommended-style section of `c4.md` is explicit:
"Boundary nesting тАж `System_Boundary` > `Container_Boundary` > `Boundary`. Functional
groups inside containers use plain `Boundary(name, "Label")`." The container wrapper
must therefore be `Container_Boundary`, with plain `Boundary` reserved for the group.

**Action.** Ran `c4_lint.py docs/C4 --format json`. Result: **178 HIGH violations of
`C4-C3-001`, spread across all 23 per-group component diagrams** (avg 7.7
components/diagram). Reading `c4_nasim_component_agent.puml:22-32` confirms the
pattern:

```plantuml
Boundary(nasim_application, "NASIM Application") {        ' тЖР should be Container_Boundary
    Boundary(agent_group, "Agent Group") <<service>> {
        Component(agent_orch, "AgentOrchestrator", ...)
```

`docs/C4/README.md:43` documents the choice ("within `Boundary(nasim_application,тАж)`")
тАФ so this is *deliberate but rule-violating*: every `Component()` ends up inside two
plain `Boundary()` blocks and zero `Container_Boundary()`, which the C4 model treats
as "no container context".

**Result.** Fix is one token per file: change the outer
`Boundary(nasim_application, "NASIM Application")` тЖТ
`Container_Boundary(nasim_application, "NASIM Application")` in all 23 component
diagrams (the inner group `Boundary(...)` stays). Clears all 178 violations.
`c4_fix.py --in-place docs/C4` should be tried first, then re-lint.

## ЁЯЯв CAR-C4-PASS тАФ C4 strengths to preserve

- Include pinned to a **stable tag** (`C4-PlantUML/v2.10.0/...`) per `cicd.md`
  ("!include must not reference `master`"). тЬУ

## ЁЯЯб CAR-SM-02 тАФ SM state palette **collides** with the CSR layer palette (root cause of CAR-SQ-03)

**Challenge.** `sm.md`: "every state must use exactly one hex colour тАж no two states
share a colour" and the colour must appear in both the SM diagram and the SQ `hnote`.
Implicitly, a colour vocabulary must be *unambiguous*.

**Action.** The SM state hexes and the CSR layer hexes are **the same codes**:

| Hex | SM state (`sm.md` / `SM/README.md`) | CSR layer (`c4_styles.puml`) |
| --- | --- | --- |
| `#FFF3E0` | THINKING | **Service** |
| `#F3E5F5` | TOOL_EXEC | **Infrastructure** |
| `#E8F5E9` | RESPONDING | **Repository** |
| `#E3F2FD` | PLANNING | **Controller** |

hnote* or a *CSR box*. This is the mechanism that produced the 77 % box-colour error
rate in [CAR-SQ-03](#-car-sq-03--csr-layer-colour-discipline-broken-418-of-541-boxes--77-high): an author reaching for "the orange one" cannot tell whether orange means
THINKING or Service.

**Result.** Split the two vocabularies. Recommended: keep the four CSR layer colours
reserved exclusively for boxes, and re-palette the agent states onto a disjoint set
(e.g. the `#FBE9E7`/`#F1F8E9`/`#FFF8E1` family already used for EVL/EDT states), or
vice-versa. Document the two palettes side-by-side in `c4_styles.puml`.

## ЁЯФ┤ CAR-SQ-01 тАФ Empty Service boxes + undeclared `svc` lifelines (7 files, CSR correctness defect)

**Challenge.** `csr.md` / `sq.md` "CSR Layering Rules": the chain
`Controller тЖТ Service тЖТ Repository` must be *visible*; "no layer is ever skipped."
Mechanically, every PlantUML lifeline used in a message **must be declared** тАФ
otherwise the renderer auto-creates it *outside every box*, which both breaks the
visual and is a latent diagram bug.

**Action.** `rod_csr_sq_lint.py` raised `RCSR-003` (CRITICAL, "must include a Service
participant") on 3 files. A deterministic declared-vs-referenced scan found the deeper
defect: **7 server/CLI files declare an *empty* Service box yet send messages to an
undeclared `svc`/`ssn_svc` lifeline.** Example `sq_srv01_list_sessions.puml:19-21,41,44`:

```plantuml
box "Service Group" #F3E5F5
end box                              ' тЖР Service box is EMPTY
...
router -> svc : SRV-01 LIST Sessions(...)   ' тЖР 'svc' is never declared as a participant
svc -> store : listSessions(...)            '   PlantUML draws it OUTSIDE the CSR boxes
```

The intro note even names the intended component ("Service(SessionService)") but
`SessionService` is never placed in the box. Affected:
`sq_srv01, sq_srv02, sq_srv03, sq_srv04, sq_srv07, sq_api05` (all `тЖТ svc`) and
`sq_cli08_list_sessions` (`тЖТ ssn_svc`).

**Result.** Declare the Service participant inside the box, e.g.
`participant "SessionService" as svc`. This simultaneously clears `RCSR-003`, removes
the undeclared-lifeline bug, and restores the CSR visual the diagram claims in its
Design note. **CRITICAL** because the diagram currently renders a CSR chain it does
not actually contain.

## ЁЯФ┤ CAR-SQ-02 тАФ `sq_cli01` declares alias `router` twice, collapsing its own CLIтЖФAPI boundary

**Challenge.** `sq.md`: a lifeline maps to exactly one C4 component (TRC-02); a
diagram must not contradict its own stated invariant. PlantUML merges two
`participant ... as X` with the same alias into one lifeline.

**Action.** `sq_cli01_process_user_input.puml` declares **`router` twice**:

```plantuml
box "CLI Interface Container" #E8F5E9
  participant "ServerRouter" as router      ' :17
...
box "API Group (Entry Gate)" #E8EAF6
  participant "ServerRouter" as router      ' :21  тЖР same alias, second box
```

Because the alias is identical, PlantUML collapses both boxes onto a single lifeline тАФ
so the "CLI Interface Container" and "API Group" visually merge, **defeating the very
invariant the diagram asserts** at `:123` ("CLI NEVER calls AgentOrchestrator
directly. ALL paths go through API"). The same file also references **undeclared
`repl` and `result`** (`:46 user -> repl`, `:52 result -> result : check for slash
command`) тАФ `result` is a return value mistakenly used as a lifeline; `repl`
(REPLSession) is the intended CLI entry but is never declared (see CAR-SQ-08).

**Result.** Give the two routers distinct aliases (e.g. `cli_router` vs `api_router`)
or model the CLI entry as `REPLSession` and route `REPLSession тЖТ ServerRouter`.
Replace `result`/`repl` self-calls with a declared lifeline. The undeclared-lifeline
scan flagged **12 files / 16 symbols** total (CLI: `repl`,`result` in cli01/02/03/05/07;
Server: `svc` in srv01-04/07/api05; `ssn_svc` in cli08) тАФ all must declare their
participants.

## ЁЯЯа CAR-SQ-03 тАФ CSR layer colour discipline broken: 418 of 541 boxes (77 %, HIGH)

**Challenge.** `sq.md`: "Box colours are project-configured тАж Follow those values; do

assigned to it." The assignment is the CSR palette in `c4_styles.puml`

**Action.** Scanned every `box "<Group>" #HEX` line in `docs/SQ`, mapped the group to
its CSR layer (per `c4_styles.puml` legend), and compared the hex. Result: **418 of
541 boxes (77 %) carry a colour that is not the canonical colour for their layer**, and
**146 occurrences use 11 off-palette hexes** that are not CSR colours at all:

| Symptom | Evidence |
| --- | --- |
| Service drawn as Controller | `Agent Group` is `#E3F2FD` (Controller blue) everywhere; should be `#FFF3E0`. |
| Repository drawn as Infra | `Tool Group` is `#F3E5F5` (Infra purple); should be `#E8F5E9`. |
| Box tinted with an **SM state** colour | `Safety Group` is `#FFF9C4` (the AWAITING_APPROVAL *state* colour); should be Service `#FFF3E0`. |
| Same group, different colour across files | `API Group` is `#E8EAF6` in `sq_agt01` but `#E8F5E9` in `sq_agt02`. |
| Off-palette colours | `#E8EAF6`├Ч48, `#EDE7F6`├Ч17, `#F1F8E9`├Ч16, `#E0F2F1`├Ч15, `#F5F5F5`├Ч15, `#FFFDE7`├Ч11, тАж |

Root cause is the palette collision in [CAR-SM-02](#-car-sm-02--sm-state-palette-collides-with-the-csr-layer-palette-root-cause-of-car-sq-03).

**Result.** After splitting the two palettes (CAR-SM-02), normalise every SQ box to the
canonical CSR colour for its group. This is a high-volume but mechanical fix тАФ a
table-driven rewrite keyed on the group name in each `box` line. It is the most
*pervasive* defect in the set and the one that most damages the "read CSR layering by
colour" affordance the design depends on.

## ЁЯЯб CAR-SQ-05 тАФ ROD: Update operations missing `update_mask` (19 files, AIP-134)

`sq_srv11_update_config, sq_agt03_update_conversation, sq_tl17_update_todo,
sq_tl22_update_plan, sq_ssn06_revert_turn`, plus the EDT `apply_*` family.

**Result.** For true partial updates (`srv11_update_config`, `tl17_update_todo`,
`tl22_update_plan`) add `update_mask`. For the EDT `apply_*` strategies and
`agt03_update_conversation` (append, not field-mask update) the rule does not really
apply тАФ add a one-line intro-note justification so the linter signal is acknowledged
rather than ignored.

## ЁЯЯб CAR-SQ-06 тАФ CSR: Repository participant absent (8 files) тАФ half real, half Integration-flow nuance

**Challenge.** `csr.md` layer contract: a data-access flow ends at a Repository; an
*Integration* flow legitimately ends at an external system (no Repository), and a
*pure-compute* flow may have neither.

**Action.** `RCSR-004` (HIGH) on 8 files. Triaged against the CSR layer contract:

| File | Verdict |
| --- | --- |
| `sq_prv02_request_chat`, `sq_prv03_stream_provider_chat` | **Not a defect** тАФ these are Integration flows (`Provider тЖТ external LLM API`); CSR explicitly has no Repository here. The linter cannot tell Integration from data-access. |
| `sq_wrl03_seek_turn`, `sq_wrl04_fork_session` | **Real** тАФ Wire Log *is* a store; a Repository/store participant (`WireReader`/`WireLog`) belongs in the chain. |
| `sq_ctx01/03/05`, `sq_edt04` | **Mixed** тАФ pure-compute pipeline steps; acceptable, but should declare the absence in the Design note. |

**Result.** Add the missing store participant for the WRL flows; annotate the
Integration/compute flows so the `RCSR-004` signal is explained rather than left open.
This finding is the clearest example of *reading the rule, not just the linter*.

## ЁЯЯб CAR-SQ-07 тАФ DRY: `ref` decomposition used in only 8 of 148 SQ files

**Challenge.** `sq.md` DRY-01..05 and `anti-patterns.md` AP-01/AP-02: extract common
sub-flows into `ref` frames; never inline a peer group's sequence. Parent
orchestrators that `<<include>>` sub-UCs (CTX-01, EDT-01, EVL-01 per the UC catalog)
must `ref` those sub-flows, not re-draw them.

**Action.** Only **8/148 files (27 `ref` blocks total)** use `ref`.
`sq_agt01_process_user_task` is the **exemplar** тАФ it delegates *everything* via `ref`
(OBS-03, AGT-03, HK-04, PRV-02, HK-05, AGT-02) and inlines nothing. The remaining 140
files inline their bodies. Many are legitimately atomic (a single file read needs no
ref), but the three documented parent-decomposition SQs (`sq_ctx01`, `sq_edt01`,
`sq_evl01`) should `ref` their sub-UCs and currently do not.

**Result.** Apply the `sq_agt01` pattern to the parent orchestrators (CTX-01 тЖТ ref
CTX-02..06; EDT-01 тЖТ ref EDT-02..10; EVL-01 тЖТ ref EVL-02..09) and to any flow that
re-draws another group's steps (e.g. `sq_cli08` inlines the session listing instead of
`ref`-ing SSN-03/API-01, per AP-02). Leaf flows may stay inline.

## ЁЯЯа CAR-SQ-08 тАФ Actor & entry-chain discipline: single `User` actor (148/148); `REPLSession` in 0 diagrams

**Challenge.** `sq.md` "Multi-Interface SQ Decomposition": CLI flows use actor
`Developer тЖТ REPLSession тЖТ AgentOrchestrator`; HTTP flows use
`HTTPClient тЖТ ServerRouter`; MCP flows use `MCPClientRuntime`. The actor identifies
*which interface* originates the flow тАФ essential for a multi-interface system.

**Action.** **All 148** SQ files use a single `actor "User"`; **0** use `Developer`,
`HTTPClient`, or `MCPClientRuntime`; **0** files declare `REPLSession` as a
participant тАФ even though the UC catalog names `REPLSession` the component-owner of
`CLI-01` and `CLI-06`. The CLI flows collapse to `User тЖТ ServerRouter`, and
`REPLSession` survives only as the *undeclared* `repl` token from CAR-SQ-02.

This is **partly a deliberate decision**: `SQ/README.md:16` documents "Single Actor:
`User` (replaces Developer + HTTP Client)" under the API-First model. But two
consequences are *not* documented and are real costs:
1. You can no longer tell from a diagram whether a flow is CLI, HTTP, or MCP тАФ which
   the multi-interface design in `design-chain.md` explicitly requires.
2. `REPLSession`, `ArgParser` (CLI Controller components in C4/UC) appear in zero SQ
   diagrams тЖТ a C4тЖФSQ coverage gap (components owned by a UC but absent from its SQ).

**Result.** Either (a) restore interface-specific actors + the REPLSession entry hop
for the 8 CLI diagrams (keeps `sq.md` conformance and C4тЖФSQ coverage), or (b) keep the
single `User` actor but **document the entry-chain deviation explicitly** in
`SQ/README.md` *and* add a note to the C4 CLI component diagram that REPLSession/ArgParser
are intentionally not shown at SQ level. Option (a) is recommended тАФ the cost of (b)
is losing interface provenance in every diagram.

## ЁЯФ╡ CAR-SQ-09 тАФ `SQ-008` (117├Ч) is mostly a linter limitation, not a defect

**Challenge.** `sq.md`: the summary note `Success:`/`Failure:` fields "MUST include
HTTP status codes **(or structured result types for internal components)**."

**Action.** `sq_lint.py` flags `SQ-008` on 117/148 files. Triaged:
- The **11 SRV (API) flows** all already carry HTTP codes *in the body*
  (`grep '200 OK\|500'` тЖТ 11/11) but not always in the summary `Success:` line тАФ a
  genuine but minor polish gap.
- The **~106 internal flows** correctly use structured result types
  (`AgentEvent(Done)`/`AgentEvent(Error)` in `sq_agt01:117-118`,
  `LLMResponse`/`ProviderError` in `sq_prv02`, `ToolResult`) тАФ which `sq.md`
  **explicitly permits**. For these, `SQ-008` is a **false positive**: the linter only
  pattern-matches `\d{3}` codes and doesn't recognise structured results.

**Result.** Echo the HTTP codes into the summary `Success:`/`Failure:` of the 11 SRV
flows (quick). For the internal flows, no change is needed тАФ instead, **improve
`sq_lint.py` SQ-008** to accept `AgentEvent(...)`/`*Result`/`*Error` as a valid
"structured result type," eliminating ~106 false positives. Treat the headline "117"
as **~11 real + ~106 noise**.

## ЁЯФ╡ CAR-SQ-10 тАФ Activation bars absent (RSQ-006, 34 files)

**Challenge.** `software-design-2026.md` SQ guidance: activation bars should show
active processing on a lifeline.

**Action.** `RSQ-006` (MEDIUM) on 34 files lacking `activate`/`deactivate`. Many leaf
flows omit them.

**Result.** Low priority; add `activate`/`deactivate` pairs for readability where a
lifeline does non-trivial work. `sq_srv01` and `sq_agt01` model these correctly тАФ use
