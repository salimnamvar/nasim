Building a terminal-native AI coding assistant requires a highly responsive architecture that carefully balances local execution, state management, and seamless LLM API compatibility.

Here is a comprehensive technical blueprint for developing a local sibling to Claude Code.

### Phase 1: Initial Setup & Technical Stack

Drawing inspiration from the architectures of Aider (Python), Cline (TypeScript), and OpenCode (Go), a **TypeScript/Node.js** stack is optimal for this application. It offers deep native alignment for managing project files within VS Code and provides robust, standardized tooling (like `npm`, `standard-version`, or `release-it`) to enforce strict semantic versioning across your software releases.

* **Real-Time Terminal Output (UI):** Use **Ink**, which allows you to build interactive CLI applications using React components. It manages complex terminal rendering, text streaming, and layout recalculations beautifully without cluttering the console.
* **Secure Shell Execution:** Utilize **Execa**. It handles child processes gracefully, executing bash commands securely while capturing `stdout`/`stderr` asynchronously so the terminal UI does not freeze during long-running tasks like test suites or server restarts.
* **File Operations:** Stick to native `fs/promises` paired with a robust diff-parsing library (like `diff`) so the AI can apply precise patch edits rather than rewriting entire files.

### Phase 2: API Integration & Environment Redirection

To allow the CLI to serve as a drop-in replacement for existing Anthropic-based tools, it must natively construct and parse payloads using the **Anthropic Messages API format** while intelligently routing requests.

* **Dynamic Endpoint Resolution:** Build your HTTP client logic to intercept standard environment variables before defaulting to the cloud.
```typescript
const baseURL = process.env.ANTHROPIC_BASE_URL || "https://api.anthropic.com/v1";
const apiKey = process.env.ANTHROPIC_API_KEY || "YOUR_DEFAULT_KEY";

```


* **Authentication Bypassing:** When redirecting to a local proxy or LM Studio, the CLI must safely accept non-empty placeholder strings for the API key. Always configuring the key to a string like `"local"` prevents the underlying Anthropic SDKs from throwing missing credential errors.
* **Model Mapping:** Use variables like `ANTHROPIC_DEFAULT_MODEL` to map cloud model requests (e.g., `claude-3-5-sonnet`) to your local variants (e.g., `qwen3-coder` or `llama-3`).

### Phase 3: The Translation Bridge & Tool Execution

The "Bridge" is an essential middleware component (which can run locally on `localhost` or a dedicated home server) that translates the Anthropic tool-use schema into generic prompts that open-source models understand.

1. **The Interception Layer:** When the CLI sends a request containing a `tools` array (defining capabilities like `read_file` or `execute_shell`), the Bridge intercepts this. If the backend is Ollama running a model that doesn't natively support Anthropic's tool schema, the Bridge translates the tools into a formatted system prompt explaining how the LLM should output JSON tool commands.
2. **The Execution Loop:**
* The local LLM outputs a text block requesting a tool execution.
* The Bridge catches this, parses the payload, and formats it into an Anthropic `tool_calls` response object.
* The Bridge sends this back to the CLI.


3. **Local Execution:** The CLI receives the `tool_calls` instruction. Before modifying the filesystem or executing a shell command via Execa, implement a "Plan Mode" (a standard in tools like OpenCode) that surfaces the requested command to the user for explicit approval (Y/n). Once executed, the CLI sends the `tool_result` back through the Bridge.

### Phase 4: Local State Management & Knowledge Repository

For maximum performance, the local knowledge repository should mirror the `~/.claude/` structure but utilize an embedded database for speed.

* **Storage Engine:** Use **SQLite** (via `better-sqlite3`) for synchronous, ultra-low latency access to local state, paired with JSON Lines (`.jsonl`) files to maintain immutable, append-only conversation logs.
* **Context & Repository Maps:** The DB schema must be designed to handle complex referential data seamlessly. When loading context for the technical documentation and architectural audit of the TENAS Model Management Service, the database must swiftly retrieve Entity Relationship Diagrams (ERDs) and API data contracts into the active prompt window without blocking the main terminal thread.
* **Session Management:** Structure the DB to index sessions by Git branch and current working directory, allowing the CLI to instantly resume the correct conversational context the moment it is launched in a specific terminal path.
