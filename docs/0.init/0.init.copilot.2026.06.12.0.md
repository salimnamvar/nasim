**Build a local, Claude‑Code–compatible terminal coding assistant with a Client‑Server LLM, Anthropic Messages API compatibility, env‑driven redirection, a local knowledge repo, and a translation Bridge.** Below is a compact, actionable blueprint you can implement on `black` (server) + `salim‑hp` (client).

### Architecture Overview
- **Client**: CLI (terminal) that speaks *Anthropic Messages* format and streams SSE for real‑time output.  
- **Server**: Local LLM host (Ollama or LM Studio) exposing an **Anthropic Messages** compatible HTTP endpoint.  
- **Bridge**: Optional translation/proxy that converts Anthropic requests to the server’s native API and implements tool calls (filesystem, shell).

### Step 1 Initial Setup Server
1. **Install Ollama or LM Studio** and bind to `0.0.0.0` so the client can reach it. Test `/v1/messages` or Ollama root.   [docs.ollama.com](https://docs.ollama.com/api/anthropic-compatibility)  [lmstudio.ai](https://lmstudio.ai/docs/developer/core/server)  
2. **Pull a code model** (e.g., `qwen3-coder`) and verify inference via curl.   [ollama.com](https://ollama.com/library/qwen3-coder)  
3. **Harden network**: firewall rules, TLS (nginx reverse proxy + cert), and mTLS if needed.   [docs.ollama.com](https://docs.ollama.com/api/anthropic-compatibility)

---

### Step 2 API Integration and Env Redirection
- **Client behavior**: send `POST /v1/messages` with `messages` array and `stream:true` for SSE. Use the Anthropic Messages schema.   [platform.claude.com](https://platform.claude.com/docs/en/api/messages)  
- **Env toggles**:
  - `ANTHROPIC_BASE_URL` → `http://<BLACK_IP>:11434` or proxy URL.  
  - `ANTHROPIC_API_KEY` → **non‑empty placeholder** (e.g., `local`) to avoid cloud auth flows.  
  - `ANTHROPIC_DEFAULT_*` → map requested Claude models to local model names.  
- **Implementation**: client reads env at startup and routes requests; fallback to official Anthropic when `ANTHROPIC_BASE_URL` unset.

---

### Step 3 Local Knowledge Repo Design
- **Root**: `~/.localcode/`  
  - `state/` — small JSON files for session metadata (indexed by UUID).  
  - `history/` — append‑only NDJSON per session for streaming replay.  
  - `context/` — LRU cache of tokenized context chunks (binary, memory‑mapped).  
  - `embeddings/` — vector DB (FAISS or SQLite+pgvector) for repo search.  
  - `locks/` — file locks for concurrent CLI sessions.  
**Performance tips**: memory‑map token caches, keep recent context in RAM, use incremental NDJSON writes for crash safety.

---

### Step 4 Tool Use and the Bridge
1. **Proxy design**: FastAPI/Express service that:
   - Accepts Anthropic Messages requests.  
   - Maps model names and streaming semantics to the LLM server.  
   - Implements **tool schema** handling: when model emits `tool_use` blocks, the proxy validates, executes in a sandboxed worker, returns tool outputs as `tool_result` blocks.   [PyPI](https://pypi.org/project/claude-proxy/)  [deepwiki.com](https://deepwiki.com/ollama/ollama/3-api-reference)  
2. **Tool sandboxing**: run filesystem/shell actions in ephemeral containers or restricted subprocesses; log and require explicit user consent for destructive ops.  
3. **Security**: auth token for proxy, audit logs, rate limits, and opt‑in telemetry.

---

### Step 5 Implementation Recommendations
- **Language**: **Python** for fastest dev (FastAPI, `rich`, `asyncio`, `subprocess`, FAISS/pgvector). Use **Node.js** if you need tight integration with existing JS tooling; **Rust** for production performance and safety. - **Key libs**: FastAPI/uvicorn, sse-starlette, prompt_toolkit/rich, FAISS or pgvector, Docker for sandboxing.  
- **Dev flow**: build CLI + local proxy, test with `curl` and Claude Code pointed at your `ANTHROPIC_BASE_URL`, iterate on tool schemas.
