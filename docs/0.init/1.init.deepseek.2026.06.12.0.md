# Building a Local LLM-Powered Terminal AI Coding Assistant: Technical Blueprint

This blueprint draws from real-world architectures and implementation patterns of existing open‑source tools to provide a structured, production‑ready plan for building a terminal-native AI coding assistant that acts as a local “sibling” to Claude Code.

## 1. Architecture Overview

Adopt a **Client‑Server (C/S) model** inspired by OpenCode and Aider, where the server runs on a powerful machine (e.g., your `black` server) and the client runs on a developer laptop (e.g., `salim-hp`).

```
┌─────────────────────────────────────────────────────────────────────┐
│                           Client (`salim-hp`)                       │
│  ┌──────────────┐    ┌────────────────┐    ┌─────────────────────┐  │
│  │   Terminal   │◄──►│  CLI Handler   │◄──►│  State Manager      │  │
│  │     TUI      │    │  (commands,    │    │  (~/.assistant/)    │  │
│  │              │    │   flags)       │    │                     │  │
│  └──────────────┘    └───────┬────────┘    └─────────────────────┘  │
│                              │                                       │
│                              │ HTTP / SSE                           │
│                              ▼                                       │
│  ┌─────────────────────────────────────────────────────────────────┐│
│  │               Environment Redirection Layer                     ││
│  │  (ANTHROPIC_BASE_URL, ANTHROPIC_API_KEY)                       ││
│  └─────────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────────┘
                                   │
                                   │ HTTP / SSE
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                           Server (`black`)                          │
│  ┌──────────────┐    ┌────────────────┐    ┌─────────────────────┐  │
│  │  LLM Engine  │◄──►│Translation    │◄──►│  Tools & Bridge     │  │
│  │  (Ollama/    │    │Proxy          │    │  (filesystem, bash, │  │
│  │   LM Studio) │    │(Anthropic ↔   │    │   LSP, git)         │  │
│  │              │    │ OpenAI-compat)│    │                     │  │
│  └──────────────┘    └────────────────┘    └─────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

### Why Client‑Server?

OpenCode’s architecture justifies this separation: the server handles core logic, LLM context management, filesystem operations and LSP interactions, while the client focuses purely on user interaction. This makes it possible to move the heavy LLM inference to a remote high‑performance server without blocking the user interface.

## 2. API Compatibility: Speaking the Anthropic Messages Format

Claude Code (and similar tools) communicate using the **Anthropic Messages API** — a specific REST + SSE format. Your assistant must speak this same protocol natively so that existing tooling can later be redirected to it.

### 2.1 Native Support

Both Ollama (since January 2026) and LM Studio 0.4.1+ provide a native `/v1/messages` endpoint that follows the Anthropic format. Llama.cpp also supports the Anthropic Messages API. This means you can implement your assistant to directly call these endpoints without any translation.

### 2.2 API Endpoints to Implement

At minimum, your assistant should expose (or be able to call) these endpoints:

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/v1/messages` | Anthropic Messages API — the core conversation endpoint |
| POST | `/v1/messages/count_tokens` | Token counting (optional but helpful) |
| GET | `/v1/models` | List available models (required for model discovery) |
| GET | `/health` | Liveness check |

Rosetta‑LLM demonstrates a well‑designed multi‑format proxy that supports all three endpoint families (`/v1/chat/completions`, `/v1/responses`, `/v1/messages`) with full streaming and lossless translation of thinking blocks and tool calls.

## 3. Environment Redirection: Toggling Between Local and Official API

This is the cornerstone of making your assistant a “drop‑in sibling”. The mechanism is **simple and battle‑tested**: setting three environment variables redirects all API traffic.

### 3.1 The Three Variables

```bash
# Redirects all API calls from api.anthropic.com to your local server
export ANTHROPIC_BASE_URL="http://<BLACK_SERVER_IP>:11434"

# Must be set to a non‑empty placeholder (local servers ignore authentication)
export ANTHROPIC_API_KEY="local"

# Map the model tiers to your actual local model names
export ANTHROPIC_DEFAULT_SONNET_MODEL="qwen3-coder"
export ANTHROPIC_DEFAULT_OPUS_MODEL="qwen3-coder"
export ANTHROPIC_DEFAULT_HAIKU_MODEL="qwen3-coder"
```

According to the official Claude Code documentation, `ANTHROPIC_BASE_URL` redirects all API calls away from Anthropic’s servers. The `ANTHROPIC_API_KEY` variable only needs to be **non‑empty** when using a local proxy; it serves as a placeholder value that satisfies authentication checks without requiring a real key.

For **model discovery**, add:

```bash
export CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1
```

This allows your assistant to query the server’s `/v1/models` endpoint and list all available local models in the `/model` picker.

### 3.2 Implementation in Your Assistant

In your code, read the environment variables at startup:

```python
# Python example
import os

ANTHROPIC_BASE_URL = os.environ.get("ANTHROPIC_BASE_URL", "https://api.anthropic.com")
ANTHROPIC_API_KEY = os.environ.get("ANTHROPIC_API_KEY", "")

if ANTHROPIC_BASE_URL != "https://api.anthropic.com":
    # We are in "local mode" — no real API key needed
    use_local_server = True
    base_url = ANTHROPIC_BASE_URL
else:
    # Use real Anthropic API with the key
    use_local_server = False
```

## 4. Local Knowledge Repo: The `~/.assistant/` Directory Structure

Inspired by Claude Code’s `.claude` directory, implement a well‑structured local state repository that maintains conversation history, context, memory, and tool configuration.

### 4.1 Directory Layout

```
~/.assistant/
├── CLAUDE.md                    # Global memory (loaded for all projects)
├── settings.json                # Global configuration
├── settings.local.json          # Local overrides (excluded from git)
├── sessions/                    # Conversation history
│   ├── <session_id_1>.json
│   ├── <session_id_2>.json
│   └── index.db                 # Session index (SQLite)
├── memory/                      # Persistent context
│   ├── project_memory.db        # Project‑scoped knowledge
│   ├── global_memory.db         # User‑scoped knowledge
│   └── embeddings/              # Vector embeddings for RAG
├── tools/                       # Tool configurations
│   ├── registered_tools.json
│   └── hooks/                   # Pre/post execution hooks
├── cache/                       # Cached data (LLM responses, file indices)
│   ├── responses/
│   └── file_index/
├── logs/                        # Structured logs
│   └── assistant.log
└── state.db                     # Main SQLite database
```

### 4.2 File Descriptions

| File/Directory | Purpose |
|----------------|---------|
| `CLAUDE.md` | Global memory file — loaded at the start of every session. Place personal coding preferences and universal instructions here. |
| `settings.json` | Global configuration (API endpoints, model defaults, permissions). |
| `sessions/` | Stores full conversation transcripts. Each session can be compressed periodically as context windows fill up. |
| `memory/` | Two‑tier persistent memory: global (user‑wide) and project‑scoped. Contents are loaded at session start to provide continuous context. |
| `state.db` | SQLite database for fast queries of session metadata, cost tracking, and event logs. |

### 4.3 Implementation Priorities

Claude Code’s documentation notes that most users only edit `CLAUDE.md` and `settings.json`; the rest of the directory is optional. Start by implementing:

1. **`CLAUDE.md` parsing** — Load this file at session start and append its contents to the system prompt.
2. **`settings.json`** — Define schema for `base_url`, `model_name`, `permissions` (allow/deny tool calls), and `hooks` for pre‑/post‑execution scripts.
3. **Session persistence** — Save each conversation turn to a session file, enabling resume and history browsing.

For maximum performance, use **SQLite** for structured metadata (sessions, tool calls, cost tracking) and **flat JSON files** for large, append‑only blobs (conversation histories). This hybrid approach avoids parsing huge JSON files for simple index queries.

## 5. Tool Use & The Translation Bridge

To enable **filesystem edits and shell commands**, your assistant must support Anthropic‑style tool use (function calling). This involves two layers: the **Client’s tool execution engine** and the **Server’s translation bridge** (if the local LLM uses a different tool‑calling format).

### 5.1 Tool Schema Definition

Define tools using Anthropic’s tool format in your system prompt:

```json
{
  "tools": [
    {
      "name": "read_file",
      "description": "Read the contents of a file",
      "input_schema": {
        "type": "object",
        "properties": {
          "path": {"type": "string"},
          "offset": {"type": "integer"},
          "limit": {"type": "integer"}
        },
        "required": ["path"]
      }
    },
    {
      "name": "write_file",
      "description": "Write content to a file",
      "input_schema": {
        "type": "object",
        "properties": {
          "path": {"type": "string"},
          "content": {"type": "string"}
        },
        "required": ["path", "content"]
      }
    },
    {
      "name": "run_command",
      "description": "Execute a shell command",
      "input_schema": {
        "type": "object",
        "properties": {
          "command": {"type": "string"},
          "cwd": {"type": "string"}
        },
        "required": ["command"]
      }
    }
  ]
}
```

### 5.2 Tool Loop Implementation

Your assistant’s core agent loop must:

1. Send user prompt + tool definitions to the LLM.
2. Parse the LLM’s response for tool calls.
3. Execute the requested tool locally (with user approval for sensitive operations).
4. Append the tool result to the conversation.
5. Loop until the LLM produces a final answer.

Aider’s `Coder` class demonstrates this central orchestration pattern: it manages message history, model communication, repository integration, and cost tracking in a unified loop.

### 5.3 The Translation Bridge (For Model Compatibility)

If your local LLM (e.g., raw Llama.cpp or older Ollama models) uses **OpenAI’s function calling format** instead of Anthropic’s tool format, you need a translation proxy between your assistant and the LLM.

Open‑source proxies that already solve this problem include:

- **Rosetta‑LLM**: A lossless bidirectional translation proxy that translates between OpenAI Chat Completions, OpenAI Responses, and Anthropic Messages formats, including thinking blocks, tool calls, and reasoning. It works as a Claude Code LLM gateway with full model picker support.
- **HydraTeams**: A translation proxy that intercepts Anthropic Messages API calls and translates them to any provider’s format. Key insight: “the teammate is still a full Claude Code instance with every tool — Read, Write, Edit, Bash, Glob, Grep, Git. It just doesn’t know its brain is GPT instead of Claude”.
- **claude-proxy** (codex‑claude‑proxy): A local proxy that translates Anthropic API requests into ChatGPT Codex calls, with support for native tool calling.

You can either integrate one of these proxy libraries directly into your server, or re‑implement a lightweight adapter that maps:

```
Anthropic tool calls  ←→  OpenAI function calls
Anthropic Messages    ←→  OpenAI Chat Completions
```

### 5.4 Bridge Implementation Plan

For minimal complexity, the translation bridge can be implemented as a separate Node.js or Python service that runs on the server and forwards requests. Run it on port 4000 (or any available port) and set the client’s `ANTHROPIC_BASE_URL` to point to it:

```bash
export ANTHROPIC_BASE_URL="http://<BLACK_SERVER_IP>:4000"
export ANTHROPIC_API_KEY="any-value"
claude
```

Rosetta‑LLM’s canonical IR approach is particularly valuable: it translates all formats to an intermediate representation, ensuring lossless conversion of thinking blocks and tool calls even when moving between providers.

## 6. Technical Stack Recommendations

Based on existing tools and real‑world performance benchmarks, here are **three viable paths** with trade‑offs.

### 6.1 Option 1: TypeScript/Node.js with Ink

**Used by:** Claude Code, Gemini CLI, and many terminal tools.

| Strengths | Weaknesses |
|-----------|------------|
| Ink library provides React‑like UI for terminal rendering | Higher memory overhead |
| Massive ecosystem of npm packages | Slower than compiled languages for high‑throughput token streaming |
| Fast development cycles | Node.js’s event loop can stall during CPU‑intensive operations |

**Key libraries:**
- `ink` (React for CLIs) — TUI rendering
- `ws` (WebSocket) — Real‑time communication
- `yargs` or `commander` — CLI argument parsing
- `sqlite3` — Local state management
- `child_process` — Shell execution (with careful sandboxing)

### 6.2 Option 2: Rust

**Used by:** OpenAI’s Codex CLI (recently migrated from TypeScript to Rust).

| Strengths | Weaknesses |
|-----------|------------|
| Eliminates runtime dependencies; single ~5MB binary | Steeper learning curve |
| Guaranteed memory safety and thread safety | Fewer terminal TUI libraries |
| Extremely fast token streaming and JSON parsing | Slower development cycles |
| Ideal for high‑volume token processing | |

**Key libraries:**
- `ratatui` — TUI framework (spiritual successor to `tui-rs`)
- `tokio` — Async runtime for non‑blocking I/O
- `reqwest` — HTTP client with SSE support
- `serde` — JSON serialization
- `rusqlite` — SQLite bindings
- `tracing` — Structured logging (used in `acp-bridge`)

The OpenAI Codex CLI team’s decision to switch from Node.js/TypeScript to Rust was driven by security and performance requirements — Rust eliminates the runtime dependencies that Node.js requires.

### 6.3 Option 3: Python

**Used by:** Aider, Rosetta‑LLM, and many LLM tools.

| Strengths | Weaknesses |
|-----------|------------|
| Rich AI/LLM ecosystem (LiteLLM, HuggingFace) | Slower than Rust for high‑throughput streaming |
| Quick prototyping and iteration | Global Interpreter Lock (GIL) limits concurrency |
| Excellent for data processing and JSON handling | Distribution requires bundling dependencies |

**Key libraries:**
- `rich` or `textual` — Terminal UI and rich formatting
- `litellm` — Unified interface to 100+ LLM providers
- `aiohttp` or `httpx` — Async HTTP with streaming
- `pydantic` — Schema validation for tools
- `sqlite3` (built‑in) — Local state
- `asyncio.subprocess` — Async shell execution

### 6.4 Recommendation: Start with Rust, Use TypeScript for Prototyping

If you are building for **production** and long‑term maintainability, **Rust** is the strongest choice. It gives you the performance needed for streaming 90+ tokens per second, eliminates runtime dependencies (crucial for terminal distribution), and provides memory safety for executing user‑provided code. The `acp-bridge` proxy demonstrates a production‑grade Rust implementation with zero runtime dependencies, retry logic with exponential backoff, and structured logging.

If you prioritize **rapid prototyping** and iterating on the user experience, start with **TypeScript/Node.js + Ink**, then rewrite performance‑critical components (token streaming, filesystem operations) in Rust as needed.

## 7. Step‑by‑Step Implementation Plan

### Phase 1: Foundation (Week 1)

1. **Set up project structure** (Rust with Cargo or TypeScript with npm).
2. **Implement environment variable parsing** (`ANTHROPIC_BASE_URL`, `ANTHROPIC_API_KEY`, `ANTHROPIC_DEFAULT_*_MODEL`).
3. **Build a basic HTTP client** to call `/v1/messages` (both streaming and non‑streaming).
4. **Implement `/v1/models` endpoint call** for model discovery.
5. **Create `~/.assistant/` directory structure** (sessions, settings, memory).

### Phase 2: Core Agent Loop (Week 2)

1. **Implement the tool definition system** (schema validation with Serde/Pydantic).
2. **Build the tool execution engine**:
   - `read_file` (with path sanitization)
   - `write_file` (with backup/rollback)
   - `run_command` (with cwd control and output capture)
3. **Implement the agent loop**:
   - Send prompt + tools → LLM
   - Parse tool calls
   - Execute tools locally
   - Append results
   - Repeat until final answer
4. **Add user permission prompts** for shell commands and file writes.

### Phase 3: Local Knowledge Repo (Week 3)

1. **Implement session persistence** — Save each conversation turn to `~/.assistant/sessions/<session_id>.json`.
2. **Add `CLAUDE.md` loading** — Read global memory file at session start.
3. **Implement `settings.json` parsing** — Allow configuring model name, base URL, and tool permissions.
4. **Build session resume capability** — Load previous session from disk and continue the conversation.

### Phase 4: Translation Bridge (Week 4)

1. **Implement or integrate a translation proxy** (Rosetta‑LLM or custom adapter).
2. **Map Anthropic tool calls ↔ OpenAI function calls**.
3. **Add support for non‑Anthropic models** (Llama.cpp, vLLM, etc.) via OpenAI‑compatible API.
4. **Test with Ollama and LM Studio**.

### Phase 5: Polish & Advanced Features (Week 5+)

1. **Add TUI interface** using `ratatui` (Rust) or `ink` (TypeScript).
2. **Implement token usage and cost tracking**.
3. **Add vector embeddings for semantic search** of `CLAUDE.md` memory.
4. **Implement multi‑agent collaboration** (inspired by OpenCode’s primary/subagent system).
5. **Add support for MCP (Model Context Protocol)** tools.

## 8. Security Considerations

- **Sandbox shell commands** — Never execute arbitrary shell commands without user confirmation. Implement an allow‑list for approved commands.
- **Path traversal protection** — Validate all file paths to prevent `../../etc/passwd` attacks.
- **Tool permission tiers** — Use Cline’s approach: allow certain tools (e.g., `read_file`) by default, require approval for others (`run_command`, `write_file`).
- **API key handling** — Never log or expose `ANTHROPIC_API_KEY`. Use environment variables exclusively.

## 9. Testing & Validation

Test your assistant against the same tooling that will eventually consume it:

1. **Set `ANTHROPIC_BASE_URL`** to your assistant’s endpoint.
2. **Run Claude Code** with these environment variables — it should work seamlessly with your assistant as the LLM backend.
3. **Verify all tools work** (`/help`, file edits, shell commands) exactly as they would with the official Anthropic API.

This is the ultimate validation that you have built a true “sibling” — not just a wrapper, but a fully compatible drop‑in replacement.

## 10. Conclusion

Building a local LLM‑powered terminal AI coding assistant requires:

1. **Client‑Server architecture** separating UI from heavy LLM inference.
2. **Native Anthropic Messages API support** for drop‑in compatibility with existing tooling.
3. **Environment variable redirection** (`ANTHROPIC_BASE_URL`, `ANTHROPIC_API_KEY`) for toggling between local and official APIs.
4. **Structured local knowledge repo** (`~/.assistant/`) with layered memory (global → project → session).
5. **Tool use engine** with a translation bridge for non‑Anthropic models.
6. **Performance‑focused stack** — Rust for production, TypeScript for prototyping.

The ecosystem has already proven that local LLMs are “good enough” for daily coding tasks. By following this blueprint, you can create a free, private, and performant assistant that works alongside Claude Code — not as a competitor, but as a local sibling.
