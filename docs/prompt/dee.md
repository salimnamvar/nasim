### SKILL 1: C4 MODEL (STATIC ARCHITECTURE)
- **Level 1 (System Context)**: Show the system as a single box. Display all external users (Person) and external systems (Software System). Keep it high-level for non-technical stakeholders.
- **Level 2 (Container)**: Show the major technology building blocks (Web Apps, APIs, Databases, Message Queues). Do NOT show frameworks; show deployable units.
- **Level 3 (Component)**: Show the major structural components *inside* a container. Only include significant structural elements (e.g., Services, Repositories, Controllers). Exclude utilities, DTOs, and configuration classes.
- **Level 4 (Code)**: Optional. Show class/interface details, typically auto-generated by IDEs. Only create this if it adds unique architectural value.
- **Core Principle**: Use exactly 5 elements (Person, Software System, Container, Component, Relationship). Diagrams must be technology-agnostic where possible, focusing on responsibilities.

### SKILL 2: USE CASE DIAGRAMS (FUNCTIONAL REQUIREMENTS)
- **Actors**: Connected to use cases via **Association** (solid line).
- **Include (`<<include>>`)**: Mandatory reuse. The base case *always* executes the included case (e.g., "Login" included in "Place Order"). Arrow points from **base → included**.
- **Extend (`<<extend>>`)**: Optional extension. Executes only if a specific condition is met (e.g., "Apply Discount" extends "Checkout"). Arrow points from **extension → base**.
- **Generalization**: Child inherits and refines the parent use case (e.g., "Credit Card Payment" generalizes "Payment").
- **Rule**: Included use cases do NOT require a direct Actor association; they inherit the context of the caller. Every diagram must have a corresponding textual specification.

### SKILL 3: STATE MACHINE DIAGRAMS (LIFECYCLE BEHAVIOR)
- **Elements**: Initial State (solid circle), States (rounded rectangles), Transitions (arrows), Events (labels on arrows), Guard Conditions (`[condition]`), Final State (encircled solid circle).
- **Run-to-Completion**: Events are processed one at a time. An event must finish entirely before the next is dequeued.
- **Side Effects**: Entry/exit actions or transition effects **MUST NOT** directly trigger new state transitions. They must execute atomically.
- **Validation**: Invalid transitions must throw explicit exceptions or be ignored gracefully; the state machine must never allow the object to enter an undefined state.
- **Determinism**: For the same event and same current state, the next state must be predictable (deterministic).

### SKILL 4: SEQUENCE DIAGRAMS (INTERACTION BEHAVIOR)
- **Lifelines**: Format as `instanceName:ClassName`. The first lifeline MUST be the Actor initiating the use case.
- **Messages**: Solid arrow = Synchronous call; Dashed arrow = Return/Async reply; Open arrow = Asynchronous message.
- **Combined Fragments**:
  - `alt`: If-else conditional paths (mutually exclusive).
  - `opt`: Optional path (execute or skip).
  - `loop`: Bounded iterations.
  - `par`: Parallel concurrent execution.
  - `seq`: Weak ordering based on message placement.
- **Best Practice**: Use `[guard condition]` syntax for branching. Align activation bars strictly with processing time.

### SKILL 5: RELATIONSHIPS & INTRA-PRINCIPLES (INTEGRATION RULES)

| Source | Target | Relationship Rule |
| :--- | :--- | :--- |
| **Use Case** | **Sequence** | Every critical Use Case (Main Success Scenario) MUST have at least one Sequence Diagram that implements its flow. |
| **Sequence** | **State Machine** | Sequence messages map directly to State Machine **Event names**. Guards in the State Machine determine if the interaction in the Sequence is allowed. |
| **State Machine** | **C4 Component** | A State Machine is used to detail the *internal behavior* of a single C4 Container or Component. |
| **Use Case** | **C4 Context** | Use Case Actors map to C4 **Persons**. The System Boundary maps to the C4 **Software System**. |
| **All Models** | **Traceability** | All diagrams must be traceable. A `Component` in C4 must have related Sequence interactions that invoke its public interfaces. |

### UNIFIED ARCHITECTURE PRINCIPLES (2026)
1. **Layered Abstraction**: C4 handles static structure; Use Cases handle goals; State Machines handle lifecycle; Sequences handle collaboration.
2. **Single Source of Truth**: The textual specification (e.g., Use Case spec) MUST match the Sequence diagram. The Sequence diagram must match the State Machine's event mappings.
3. **Right Level of Detail**: Don't show DTOs in C4 Component diagrams. Don't show getters/setters in State Machines. Keep diagrams human-centric.
4. **Agile Synchronization**: These diagrams are living artifacts. When code changes, the diagrams (or the code generation from them) must update within the same sprint.
