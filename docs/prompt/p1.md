# Audit of nasim Use Case Diagrams

## Executive Summary

The use case model is **comprehensive and well-aligned** with the C4 architecture. All 18 groups and their key operations are represented, with 143 use cases covering everything from CLI interaction to wire logging. The diagrams are well-structured, consistent in styling, and use include/extend relationships appropriately.

However, **several consistency and correctness issues** were identified:

| Issue Type | Count | Severity |
|------------|-------|----------|
| Actor misassignment (external actor for internal UCs) | 7 groups | **High** |
| Overview diagram inconsistency (wrong UC IDs/names) | 3 groups | **High** |
| Duplicate/overlapping UC definitions | 2 pairs | Medium |
| Missing UC for OTel export | 1 | Low |

---

## Detailed Issue Breakdown

### 1. Incorrect Actors for Internal Use Cases (High)

Many use cases in internal groups (Provider, Router, RepoIntelligence, Evaluation, EditStrategy, Context) are incorrectly assigned to `HTTP Client` as primary actor. These groups are **not directly exposed** to external clients; they are invoked by the **Agent** (or by the system itself) during the agentic loop.

| File | UC(s) | Current Actor | Should Be |
|------|-------|---------------|-----------|
| `uc_provider.puml` | PRV‑02, PRV‑03 | HTTP Client | **Agent** (or no actor) |
| `uc_router.puml` | RTG‑01, RTG‑02, RTG‑03 | (none) | **Agent** (or no actor) |
| `uc_repo_intelligence.puml` | RIM‑06 | HTTP Client | **Agent** (or no actor) |
| `uc_evaluation.puml` | EVL‑01 | HTTP Client | **Agent** (or no actor) |
| `uc_edit_strategy.puml` | EDT‑01 | HTTP Client | **Agent** (or no actor) |
| `uc_context.puml` | CTX‑01 | HTTP Client | **Agent** (or no actor) |
| `uc_tools.puml` | TL‑12 (DISPATCH MCP Extension) | Developer | **Agent** (internal tool) |

**Rationale:** According to C4, `HTTP Client` interacts only with the `Server` container, which then delegates to `AgentOrchestrator`. The Provider, Router, RepoIntelligence, etc. are **core library components** that are called by the Agent, not by external actors.

**Action:** Remove or replace these actors. Use `actor Agent` (add a system actor) or leave them actor‑less (use cases can be internal without a primary actor).

---

### 2. Overview Diagram Inconsistencies (High)

The `uc_overview.puml` summarises each group with one representative use case, but **the UC IDs/names do not match the detailed diagrams** in three cases:

| Group | Overview UC | Correct UC (from inventory) |
|-------|-------------|-----------------------------|
| **Provider (PRV)** | `PRV‑01 REQUEST Chat` | `PRV‑01 REGISTER Provider` (REQUEST Chat is PRV‑02) |
| **Server (SRV)** | `SRV‑01 SERVE HTTP API` | `SRV‑01 LIST Sessions` (no generic "SERVE" UC exists) |
| **Hooks (HK)** | `HK‑01 DISPATCH Hook` | `HK‑01 REGISTER Hook` (DISPATCH hooks are HK‑02 to HK‑05) |

This breaks traceability between the overview and the detailed use case set.

**Action:** Update `uc_overview.puml` to use the correct UC IDs and names as per the inventory (README.md). For Server, either use `SRV‑01 LIST Sessions` or consider adding a new `SRV‑00 SERVE HTTP API` UC if you want a high‑level summary, but ensure consistency.

---

### 3. Duplicate/Overlapping Use Cases (Medium)

- **AGT‑05 CHECK Permission** and **SAF‑01 CHECK Permission** – these are conceptually identical. The Agent’s permission check should delegate to the Safety group. Currently, both exist as separate UCs. This could lead to confusion.
  - **Suggested fix:** Remove `AGT‑05` and have `AGT‑02` (DISPATCH Tool Call) include `SAF‑01` (or use an `<<include>>` from AGT‑02 to SAF‑01). Or rename AGT‑05 to "DELEGATE Permission Check" to clarify it’s a delegator, but the duplication remains.

- **PRV‑04 SELECT Provider** and **RTG‑01 SELECT Model** – these overlap because selecting a model implicitly selects a provider (via litellm). The distinction is subtle. PRV‑04 might be about registering/selecting a provider backend, while RTG‑01 is about choosing a model string. To avoid confusion, rename PRV‑04 to "SELECT Provider Backend" and RTG‑01 to "SELECT Model" and ensure they are linked (RTG‑01 may use PRV‑04).

**Action:** Clarify the responsibilities and possibly merge or rename.

---

### 4. Missing Use Case: OTel Export (Low)

The C4 observability diagram includes an `OTelExporter` component (optional, behind a feature flag) that exports traces/metrics via OTLP. This is **not represented** in any use case. While optional, it is a system behaviour that should be captured.

**Action:** Add `OBS‑06 EXPORT OTLP` to `uc_observability.puml` (and update the inventory/overview).

---

### 5. Minor Naming Issues

- `AGT‑11 DELEGATE Task` – unclear what it delegates. According to description ("Assign tasks to specialized roles"), it relates to PersonaManager. Rename to `DELEGATE to Persona` or `ASSIGN Task to Role`.
- `TL‑12 DISPATCH MCP Extension` – actor should be Agent, not Developer.
