## 2. Remaining Issues & Improvement Opportunities

Despite the excellent progress, the following points still require attention:

### 2.1 **Outdated README Inventory** (Medium)

The `README.md` component inventories for **Edit Strategy** and **Evaluation** are not aligned with the diagrams:
- Edit Strategy README lists **7** components, but the diagram now has **10** (added FencedBlockCoder, FunctionLevelCoder, InlinePatchCoder).
- Evaluation README does **not** mention `QualitySignal`.
- Sandbox README does **not** mention `ResourceLimiter`.

**Action**: Update the README tables to include all components.

### 2.2 **Misleading Relationship in Observability Diagram** (Low)

`DualOutputAdapter` has a relationship:
```
Rel(dual_output_adapter, ContainerRuntime, "writes human-readable output when isatty")
```
This is conceptually incorrect. `ContainerRuntime` (Docker/Kubernetes) does **not** receive human‑readable output directly; it captures **stdout** regardless of content. The correct description should be:
- `Rel(dual_output_adapter, ContainerRuntime, "writes JSON to stdout (machine-readable)")`
- The human‑readable part is already delegated to `cli_renderer`, which ultimately writes to stdout as well, but the relationship to `ContainerRuntime` should represent the **emit‑only stdout stream**, not the human‑readable format.

**Action**: Reword the relationship or split into two: one for JSON emission and one for rendering delegation.

### 2.3 **Observability Platform Still Tagged as “Future”** (Medium)

In the context diagram, the relationship to `obs_platform` uses `$tags="future_rel"`. However, observability is now a core architectural pillar (tenas pattern) and should not be marked future. The platform (Loki/Prometheus/Grafana) is considered external but not “future scope”. Remove the `future` tag to reflect that it is a production‑grade design, even if deployment is optional.

**Action**: In `c4_nasim_context.puml`, replace `$tags="future_rel"` with a normal relationship (or keep but remove the “Future scope” legend entry for that relationship).

### 2.4 **Missing `CompactionPolicy` Component** (Low / Enhancement)

Earlier audit suggested adding a `CompactionPolicy` to the Agent group to explicitly define compaction rules (token threshold, message age, importance). This is a nice‑to‑have that improves clarity but is not critical.

**Action** (optional): Add `Component(compaction_policy, "CompactionPolicy", "", "Determines when/how to compact")` and a relationship from `ConversationHistory` to it.

### 2.5 **Missing `StrategyHeuristics` in Edit Strategy** (Low / Enhancement)

`StrategySelector` currently has no explicit representation of the heuristics it uses (edit size, risk, file type). Adding a `StrategyHeuristics` component would clarify the decision logic.

**Action** (optional): Add `Component(strategy_heuristics, "StrategyHeuristics", "", "Decision rules for strategy selection")` and connect to `StrategySelector`.

### 2.6 **Provider Diagram Still Has a `future` Tag for Some Elements** (Minor)

The provider diagram includes `AddElementTag("future", …)` and `AddRelTag("future_rel", …)` but no elements use them. This clutters the diagram. Remove these definitions if unused.
