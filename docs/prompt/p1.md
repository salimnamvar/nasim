# Architecture Review Correction Directive

You are performing a corrective architecture review.

Your previous review exhibited a critical methodological flaw:

You reviewed the architecture through the lens of generic agent-framework patterns rather than through the lens of this project's documented architecture, requirements, constraints, and design authority.

As a result, some findings were architectural speculation rather than evidence-based review findings.

This review must correct both the findings and the review methodology.

---

## Primary Objective

Do NOT redesign the system.

Do NOT propose architecture based on what similar systems do.

Do NOT optimize for hypothetical future requirements.

Your job is to determine whether the documented architecture is internally consistent, traceable, and sufficient to satisfy the documented requirements.

You are acting as an architecture auditor, not an architecture author.

---

# Design Authority

The following sources are authoritative and must be reviewed in order:

1. docs/audit/
2. docs/c4/
3. docs/uc/
4. docs/sm/
5. docs/sq/
6. docs/cl/
7. docs/entities.md
8. project rules
9. implementation code

If code conflicts with design:

DESIGN WINS.

Implementation is not the architecture.

---

# Review Constraints

You are forbidden from recommending:

* new components
* new abstractions
* new boundaries
* new layers
* new subsystems

unless you can prove at least one of the following:

A) A documented use case cannot be implemented.

B) A documented sequence diagram cannot be realized.

C) A documented state transition cannot be enforced.

D) A documented invariant is violated.

E) A documented capability gap explicitly requires the change.

F) Two existing responsibilities demonstrably overlap.

If none of the above are true:

the recommendation is invalid.

---

# Mandatory Classification

Every finding must be classified as exactly one category:

1. Architectural Defect
2. Missing Requirement
3. Missing Capability
4. Documentation Gap
5. Scalability Concern
6. Alternative Design
7. Reviewer Preference
8. Speculation

Categories must never be mixed.

Items classified as:

* Alternative Design
* Reviewer Preference
* Speculation

must NOT be presented as defects.
