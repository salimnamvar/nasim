# 🏗️ Technical Blueprint: Local Terminal-Native AI Coding Assistant

## Phase 1: Architecture & API Compatibility
The system follows a strict **Client-Server** model to ensure the heavy lifting (LLM inference) is decoupled from the user interface.

*   **Server (`black`)**: Hosts the LLM via **Ollama** (recommended for systemd background execution) or **LM Studio**. It must be configured to listen on `0.0.0.0` (e.g., `OLLAMA_HOST=0.0.0.0:11434`) and have firewall rules opened (`sudo ufw allow 11434/tcp`).
*   **Client (`salim-hp`)**: A terminal-native CLI application. It constructs prompts, manages local state, and renders the TUI (Text User Interface).
*   **API Compatibility**: The Client must construct HTTP POST requests to `/v1/messages` that strictly adhere to the **Anthropic Messages API schema**. This includes formatting `system` prompts, `user`/`assistant` message roles, and `tool_use`/`tool_result` content blocks. This ensures the client can act as a drop-in replacement for any tool expecting Anthropic's format.

## Phase 2: Environment Redirection System
To allow seamless toggling between the official Anthropic cloud API and your local server, the client must read and prioritize environment variables. 

Implement a configuration loader in your client that checks for these variables in this order of precedence:
1. `ANTHROPIC_BASE_URL`: Overrides the default `https://api.anthropic.com`. 
   - *Local*: `http://<BLACK_SERVER_IP>:11434` (Direct to Ollama) or `http://<BLACK_SERVER_IP>:4000` (Via Proxy).
   - *Cloud*: Leave unset or set to default.
2. `ANTHROPIC_API_KEY`: **Mandatory**. Must be set to a non-empty placeholder string (e.g., `"local"`). If unset or empty, the client will attempt browser-based OAuth or fail authentication.
3. `ANTHROPIC_DEFAULT_SONNET_MODEL`, `ANTHROPIC_DEFAULT_OPUS_MODEL`, `ANTHROPIC_DEFAULT_HAIKU_MODEL`: Map these to your local model name (e.g., `"qwen3-coder"`). This prevents "model not found" errors when the client requests a specific tier.
4. `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1`: (Optional) Enables the client to ping `/v1/models` at startup to populate a local model picker.

## Phase 3: Local Knowledge Repo Structure (`~/.local-ai-assistant/`)
For maximum performance, avoid heavy ORMs. Use a hybrid approach: **SQLite** for relational metadata/history and **flat files** for raw context.

```text
~/.local-ai-assistant/
├── config.json             # User preferences, default model, API endpoint toggles.
├── state.db                # SQLite database. 
│   ├── Table: sessions     # id, project_path, created_at, total_tokens
│   ├── Table: messages     # id, session_id, role, content_summary, timestamp
│   └── Table: tool_logs    # id, session_id, tool_name, execution_time, success (bool)
├── context/                # Flat files for fast retrieval.
│   ├── global_memory.md    # Cross-project user preferences (e.g., "Always use TypeScript").
│   └── projects/           # Directory named by project hash/path.
│       └── <project_hash>/ 
│           ├── repo_map.json # Cached file tree and symbol map for fast context injection.
│           └── active_context.md # Summarized current task state.
└── tools/                  # Custom tool definitions and sandboxed execution scripts.
    ├── read_file.sh
    └── run_command.sh
```
*Performance Tip*: Use SQLite with `WAL` (Write-Ahead Logging) mode enabled for concurrent, non-blocking reads/writes during active terminal sessions.

## Phase 4: Tool Use & The Bridge (Translation Proxy)
Local models (like `qwen3-coder` via Ollama) may not natively support the complex, multi-turn tool-calling schema of Anthropic. A **Translation Proxy** (similar to `claude-proxy`) acts as the "Bridge" to handle this.

**Proxy Implementation Plan:**
1. **Intercept**: The Proxy runs on the Server (`black`) on port `4000`. It listens for `POST /v1/messages` requests from the Client.
2. **Parse & Translate**: It parses the Anthropic-formatted `tools` array and translates it into the local LLM's native tool-calling format (e.g., Ollama's `tools` JSON schema).
3. **Execution Loop**: 
   - If the local LLM generates a tool call (e.g., `execute_shell_command`), the Proxy intercepts it *before* sending the response to the Client.
   - The Proxy executes the command securely on the Server (or relays the execution request back to the Client for local execution, depending on security requirements).
4. **Format Result**: The Proxy takes the stdout/stderr of the executed command and formats it into an Anthropic-compatible `tool_result` content block.
5. **Resume Generation**: The Proxy injects this `tool_result` back into the conversation history and sends a follow-up request to the local LLM to generate the final text response, which is then streamed back to the Client.

## Phase 5: Technical Stack Recommendations
Based on the architectures of Aider, Cline, and OpenCode, here are the top recommendations for building the **Client**:

| Language | TUI Library | Shell Execution | Pros | Cons |
| :--- | :--- | :--- | :--- | :--- |
| **Node.js / TypeScript** (Recommended) | `Ink` (React for CLI) or `Blessed` | `execa` or `node-pty` | Native alignment with Anthropic's JS SDK; massive ecosystem; `claude-proxy` is built in Node, ensuring schema compatibility. | Higher memory footprint than Rust/Go. |
| **Python** | `Rich` + `Prompt Toolkit` | `asyncio` + `subprocess` | Excellent AI ecosystem (`litellm` for universal API routing); rapid prototyping; Aider is built in Python. | Can be slower in rendering complex, high-frequency TUI updates. |
| **Rust** | `Ratatui` + `Crossterm` | `tokio` + `std::process::Command` | Maximum performance, minimal memory usage, and highly secure process sandboxing (like OpenCode's high-volume goals). | Steeper learning curve; manual JSON schema mapping for Anthropic API. |

**Recommendation**: Start with **Node.js/TypeScript**. It allows you to directly reuse Anthropic's official TypeScript types for request validation, and libraries like `execa` make secure, asynchronous shell execution with timeout handling trivial.

## Phase 6: Initial Setup & Integration Steps

1. **Server Setup (`black`)**:
   - Install Ollama: `curl -fsSL https://ollama.com/install.sh | sh`
   - Configure network access: `sudo systemctl edit ollama` → Add `Environment="OLLAMA_HOST=0.0.0.0:11434"` → `sudo systemctl restart ollama`.
   - Open firewall: `sudo ufw allow 11434/tcp`.
   - Pull model: `ollama pull qwen3-coder`.

2. **Bridge/Proxy Setup (`black`)**:
   - Install the proxy: `npm install -g claude-proxy`
   - Start the proxy: `claude-proxy start` (Runs on port `4000` by default, handling the tool-use translation).

3. **Client Setup (`salim-hp`)**:
   - Initialize your project and install dependencies (e.g., `npm init -y`, `npm install ink react`).
   - Configure your shell (`~/.bashrc` or `~/.zshrc`):
     ```bash
     export ANTHROPIC_BASE_URL="http://<BLACK_SERVER_IP>:4000"
     export ANTHROPIC_API_KEY="local"
     export ANTHROPIC_DEFAULT_SONNET_MODEL="qwen3-coder"
     export ANTHROPIC_DEFAULT_OPUS_MODEL="qwen3-coder"
     export ANTHROPIC_DEFAULT_HAIKU_MODEL="qwen3-coder"
     ```
   - Source the file: `source ~/.bashrc`.

4. **Development & Testing**:
   - Build the TUI to capture user input.
   - Construct the HTTP request to `http://<BLACK_SERVER_IP>:4000/v1/messages` with the Anthropic schema.
   - Test basic chat, then test tool use (e.g., ask the AI to "list the files in the current directory" and verify the proxy executes it and returns the result).
   - Implement the local `~/.local-ai-assistant/` SQLite logging to track token usage and session history.

This blueprint provides a robust, scalable foundation for a private, cost-free, and highly capable local AI coding assistant that mirrors the architecture of modern agentic tools.
