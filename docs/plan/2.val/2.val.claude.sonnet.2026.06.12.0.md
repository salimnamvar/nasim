# Nasim — Architecture Validation

Validation of the engineering blueprint against five targeted questions.
One structural correction required. All other decisions hold.

---

## Critical Correction: Tool Execution Boundary

**Question 1** reveals a flaw in the blueprint that must be resolved before Phase 2.

The blueprint puts `tool_executor.py`, `security.py`, and `loop.py` inside the
Bridge on `black`. That means `read_file`, `write_file`, and `bash` operate on
`black`'s filesystem. The user's codebase is on `salim-hp`. This is wrong.

Section 13 of the blueprint states the correct principle but only applies it to
Claude Code:

> "Claude Code's tools continue to execute on salim-hp as normal — Claude Code
> handles those itself. The Bridge only handles the LLM inference routing. This is
> the right division."

Apply the same principle to Nasim. The Bridge is a pure inference proxy. The
agentic loop and tool executor run on the Client, on `salim-hp`.

### The latency argument does not hold

The blueprint justifies running the loop in the Bridge to avoid "3-30 seconds
of latency per turn." This conflates two different costs:

- **Inference time:** 1–30 seconds per Ollama call on a local GPU. Unavoidable.
- **SSH tunnel overhead:** ~1–2 ms LAN latency per hop.

With loop-in-bridge: 3–10 LLM calls stay on `black`'s LAN. Tunnel overhead: 1×
(one request + one streaming response for the entire turn).

With loop-in-client: 3–10 LLM calls cross the SSH tunnel. Tunnel overhead: 10×
2ms = 20ms extra per turn.

20ms is not 3–30 seconds. The inference time dominates entirely. The latency
argument justifies collocating the loop with the model only if the per-hop
latency were measurable — it is not. Loop-in-client adds 20ms over 3–10 inferences
that each take 1–30 seconds. This does not register.

### Corrected architecture

```
salim-hp                                       black
──────────────────────────────────             ──────────────────
CLI Client (nasim)                             Bridge (:8080)
  ├── tool_executor.py  ← executes here          ├── server.py
  ├── security.py       ← enforced here          └── translator.py (optional)
  └── loop.py           ← runs here                      │
                               │                          │
                               │  per LLM call            │
                               └──────────────────────────►
                                    SSH tunnel             │
                               ◄──────────────────────────┘
                                    streaming response    Ollama (:11434)
```

The Bridge collapses to two files: `server.py` (FastAPI proxy) and `translator.py`
(Anthropic ↔ OpenAI format mapping, needed only if Ollama's native endpoint has
gaps). Nothing else runs on `black` except the LLM.

### Confirmation gate simplification

When the loop runs in the Client, the confirmation gate does not need the SSE
protocol. The loop is executing synchronously in the terminal process. A
destructive tool call pauses the loop and prompts inline:

```
⚠ bash: git push --force origin main  [risk: high]
Execute? (y/n): _
```

The SSE `confirmation_required` event design was load-bearing only because the
loop was in the Bridge (where there is no terminal to prompt). With loop-in-client,
it becomes a simple blocking `input()` in the terminal. The SSE event is not needed.
Remove it — it was architectural complexity solving a problem that no longer exists.

## Q2 — Ollama Native /v1/messages Endpoint

The blueprint's call is correct: test before building.

In mid-2026, Ollama's native `/v1/messages` endpoint is production-ready for
basic and tool-call use. The cases where gaps appear:

| Scenario | Status |
|---|---|
| Single tool call, structured JSON input | Reliable |
| Parallel tool calls in one response | Intermittent — some models emit malformed JSON |
| `stop_reason: tool_use` vs `end_turn` | Correct for qwen2.5-coder:32b |
| Tool result round-trip (tool_result content block) | Reliable |
| Streaming + tool calls combined | Reliable for Ollama ≥ 0.6 |

Run the tool-call test from section 3 of the blueprint before Phase 2. If
qwen2.5-coder:32b returns a well-formed `tool_use` block with correct `input`
JSON, use the native endpoint. The translation layer stays in Phase 4.

The only scenario where the translation layer becomes a Phase 1–2 concern: if
Ollama's `/v1/messages` drops the `tool_use` block entirely and falls back to
text describing the tool call. Test for this. It is the single failure mode that
would block the agentic loop.

## Q3 — SSE Confirmation Gate Concurrency

**This question is moot** given the correction in Q1. With loop-in-client, there
is no SSE confirmation gate. The prompt is inline in the terminal. Concurrency
is not a concern — each terminal session is its own process with its own loop.

If the Bridge-in-loop design were retained (not recommended), the SSE protocol
would be concurrent-safe because both Bridge and Client are stateless between
requests. Session context travels with each HTTP request body. Two concurrent
sessions would produce two independent SSE streams; approvals would route
correctly because the session ID in the request matches the tool_use_id in the
conversation thread. The protocol holds. But the architecture does not, for the
reasons in Q1.

## Q4 — CLAUDE.md as Sync Surface

Correct decision. No migration engine is needed.

`state.db` in Claude Code is an internal implementation detail of the Electron/
React rendering layer — it stores UI state, not conversation semantics. It is not
a session transcript. Writing to it from outside would corrupt Claude Code's
internal state. Nothing of value is lost by not reading it.

CLAUDE.md is the right sync surface because it is:
- Human-readable and human-editable
- Version-controlled (committed to the repo or `~/.nasim/`)
- Stable — Claude Code has read CLAUDE.md since its first public release
- Already the established convention for persisting project context

**One gap to address:** CLAUDE.md resolution must replicate Claude Code's
directory-walking behaviour. Claude Code reads CLAUDE.md files from the project
root and all parent directories up to `~`. Nasim must do the same — not just
`~/.nasim/CLAUDE.md`, but also `{cwd}/CLAUDE.md`, `{cwd}/../CLAUDE.md`, etc.,
merged in order from root to leaf. If Nasim only reads the global file, it will
miss project-scoped context that Claude Code injects automatically.

This is implementable in under 20 lines and should be in Phase 1, not Phase 3.

**The JSONL history is NOT a migration problem.** It is Nasim's own audit log.
Claude Code does not need to read it. Nasim's search over JSONL is a separate
capability — useful, but unrelated to Claude Code interoperability.

## Q5 — Claude Code Capability Parity Without Source Access

Claude Code's source is available on npm (`@anthropic-ai/claude-code`) and on
GitHub. It does not need to be read — Nasim is not a fork and does not depend on
it. The Anthropic Messages API is documented. The tool interface is documented.
CLAUDE.md behaviour is documented. Streaming SSE format is documented.

What Nasim can reproduce fully:

| Capability | Method |
|---|---|
| Tool loop (read, write, edit, bash, glob, grep) | Define same tools; run same loop logic |
| CLAUDE.md memory injection | Walk dirs; merge; prepend as system prompt |
| Streaming TUI with markdown + syntax highlighting | `rich` + `anthropic.messages.stream()` |
| Plan mode | System-prompt mode; `--plan` flag prepends planning instructions |
| Session history and search | JSONL + SQLite FTS5 |
| Model routing | `ANTHROPIC_BASE_URL` redirection; model map |
| Drop-in backend for Claude Code itself | Expose `/v1/messages`; Claude Code sets `ANTHROPIC_BASE_URL` |

What Nasim will not have, and does not need:

| Capability | Why it does not matter |
|---|---|
| Git worktrees for agent isolation | Claude Code uses these for concurrent agents. Nasim is single-user, single-session. Not needed in Phase 1–3. |
| IDE integration (VS Code diff panel) | Claude Code's VS Code extension hooks into Claude Code specifically. Nasim's terminal diff renderer via `rich` is sufficient. |
| MCP server protocol | Claude Code supports MCP as a tool extension mechanism. Nasim can add this in Phase 4 if needed; it is not blocking. |
| claude.ai web session sync | Irrelevant for a local-only tool. |

**The one non-obvious gap:** Claude Code's `Edit` tool uses a precise string
replacement algorithm — it requires the exact `old_string` to appear exactly once
in the file. If Nasim's `write_file` is a full-overwrite tool rather than a
surgical replacement tool, the LLM will produce larger diffs and be more prone
to introducing drift in untouched sections of the file. Add an `edit_file` tool
with `old_string` / `new_string` semantics in Phase 2 alongside `write_file`.
This is the single tool-interface decision with real quality impact.

## Validated Architecture

The architecture is sound with one structural correction and one tool addition.

**What changes:**

1. Loop, tool executor, and security policy move from the Bridge to the Client.
   Bridge becomes a pure proxy (server.py + optional translator.py).
2. Confirmation gate becomes an inline terminal prompt. SSE `confirmation_required`
   event is removed.
3. CLAUDE.md directory walking is added to Phase 1 (not Phase 3).
4. `edit_file(old_string, new_string)` tool added to Phase 2 tool definitions.

**What holds:**

- Three-component architecture: Client (salim-hp) → Bridge (black:8080) → Ollama
  (black:11434)
- SSH tunnel with no exposed ports
- `ANTHROPIC_BASE_URL` redirection, no SDK patching
- Ollama native `/v1/messages` tested first; translation layer deferred to Phase 4
- JSONL primary storage; SQLite FTS5 as read-only search cache
- No migration engine; no state.db dependency
- Python 3.12+ across the board
- 25-iteration loop cap
- Phase delivery order unchanged (Foundation → Tooling → Knowledge → Polish)

The corrected architecture is simpler than the original: the Bridge has ~100
lines of logic instead of ~400, and tool execution is colocated with the user's
filesystem where it belongs.
