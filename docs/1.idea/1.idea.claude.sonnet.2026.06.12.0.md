# Nasim — Engineering Blueprint

A terminal-native AI coding assistant. Privacy-first local sibling to Claude Code,
backed by Ollama on `black`, speaking the Anthropic Messages API natively.

---

## 1. Architecture

Three components. Hard boundary between each.

```
salim-hp                          black
─────────────────────             ─────────────────────────────────────
CLI Client (nasim)                Bridge Server (:8080)
  │                                 │  ← accepts Anthropic Messages format
  │  SSH tunnel                     │  ← runs the agentic tool loop
  │  localhost:8080 → black:8080    │  ← executes tools locally on black
  └─────────────────────────────────►  ← translates to OpenAI format
                                    │
                                    ▼
                                  Ollama (:11434)
                                    ← qwen2.5-coder:32b or qwen3-coder
```

**Why the Bridge runs on `black`, not `salim-hp`:**
The agentic tool loop calls the LLM multiple times per user turn (tool call → result →
re-submit). Collocating the Bridge with Ollama eliminates those round-trip latencies
from the loop. The only network hop is Client → Bridge, which carries one request and
one streaming response per user turn.

**Why SSH tunnel, not raw LAN HTTP:**
No firewall rules, no TLS certificate management, no exposed port. One alias handles
connectivity. `salim-hp` has no credentials for `black`'s LLM — the tunnel isolates
access to authenticated SSH sessions only.

```bash
# ~/.bashrc on salim-hp
alias nasim='ssh -f -N -L 8080:localhost:8080 black && ANTHROPIC_BASE_URL=http://localhost:8080 ANTHROPIC_API_KEY=local nasim chat'
```

## 2. Stack

**Python 3.12+ across the board.** No Rust, no Node.js.

Rationale: Salim's existing stack. The Anthropic SDK is Python-first with native
`ANTHROPIC_BASE_URL` support. `rich` + `typer` produce Claude Code-quality terminal
output in under 300 lines. The only reason to choose Rust is distributing a no-dependency
binary to end users — irrelevant for a personal tool.

| Layer | Library | Reason |
|---|---|---|
| CLI | `typer` | Declarative arg parsing, auto-generated `--help` |
| Terminal render | `rich` | Markdown, syntax highlight, live spinners, diff panels |
| Anthropic client | `anthropic` | Native SDK; reads `ANTHROPIC_BASE_URL` transparently |
| Bridge HTTP server | `fastapi` + `uvicorn` | Async-native, SSE streaming, minimal overhead |
| Outbound HTTP | `httpx` | Async client with streaming; works with Ollama and LM Studio |
| Config | `pyyaml` | Human-editable; version-controlled |
| History store | `sqlite3` (stdlib) | FTS5 search; zero extra dependency |
| Shell execution | `asyncio.subprocess` | Async; integrates with uvicorn event loop |
| Testing | `pytest` + `respx` | `respx` mocks `httpx` calls without a live LLM |

## 3. Ollama's Native Anthropic Endpoint

**Check this before building custom translation.**

Since early 2026, Ollama exposes `POST /v1/messages` natively. Test it first:

```bash
curl http://black:11434/v1/messages \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen2.5-coder:32b",
    "max_tokens": 1024,
    "messages": [{"role": "user", "content": "write a python hello world"}]
  }'
```

If tool calling works correctly end-to-end, the Bridge is a thin pass-through — no
translation logic needed. Only build the OpenAI translation layer if tool-call
fidelity is insufficient (missing `input_schema`, mangled tool names, etc.).

Test tool calling with:

```bash
curl http://black:11434/v1/messages \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen2.5-coder:32b",
    "max_tokens": 1024,
    "tools": [{"name": "read_file", "description": "Read a file", "input_schema": {"type": "object", "properties": {"path": {"type": "string"}}, "required": ["path"]}}],
    "messages": [{"role": "user", "content": "read src/main.py"}]
  }'
```

If this returns a well-formed `tool_use` content block, the translation layer is
unnecessary complexity.

## 4. API Translation (when needed)

The Bridge translates the Anthropic Messages format ↔ OpenAI Chat Completions
format only when Ollama's native endpoint is insufficient.

### Request: Anthropic → OpenAI

```python
def anthropic_to_openai(req: AnthropicRequest) -> dict:
    messages = []
    if req.system:
        messages.append({"role": "system", "content": req.system})
    for msg in req.messages:
        if isinstance(msg.content, list):
            text = " ".join(b.text for b in msg.content if b.type == "text")
        else:
            text = msg.content
        messages.append({"role": msg.role, "content": text})

    tools = None
    if req.tools:
        tools = [
            {"type": "function", "function": {"name": t.name, "description": t.description, "parameters": t.input_schema}}
            for t in req.tools
        ]

    return {
        "model": MODEL_MAP.get(req.model, req.model),
        "messages": messages,
        "max_tokens": req.max_tokens,
        "stream": req.stream,
        **({"tools": tools} if tools else {}),
    }
```

### Response: OpenAI → Anthropic

```python
def openai_to_anthropic(oai_resp: dict) -> AnthropicMessage:
    choice = oai_resp["choices"][0]
    content = []

    if choice["message"].get("content"):
        content.append({"type": "text", "text": choice["message"]["content"]})

    for tool_call in choice["message"].get("tool_calls") or []:
        content.append({
            "type": "tool_use",
            "id": tool_call["id"],
            "name": tool_call["function"]["name"],
            "input": json.loads(tool_call["function"]["arguments"]),
        })

    stop_reason = "tool_use" if any(b["type"] == "tool_use" for b in content) else "end_turn"

    return AnthropicMessage(
        id=f"msg_{uuid4().hex}",
        type="message",
        role="assistant",
        content=content,
        stop_reason=stop_reason,
        usage=Usage(
            input_tokens=oai_resp["usage"]["prompt_tokens"],
            output_tokens=oai_resp["usage"]["completion_tokens"],
        ),
    )
```

## 5. Agentic Tool Loop

The loop runs entirely inside the Bridge. The Client sees a single streaming response.

```
Client: POST /v1/messages {messages, tools}
             │
             ▼
Bridge: forward to Ollama
             │
             ▼
    ┌─ LLM response has tool_use? ─┐
    │                              │
   No                             Yes
    │                              │
    ▼                              ▼
stream back to client     check security policy
                                   │
                          ┌────────┴─────────┐
                          │                  │
                     blocked           allowed (or confirmed)
                          │                  │
                     stream error      execute tool
                                            │
                                       append tool_result
                                       re-submit to LLM (loop)
                                       (cap: 25 iterations)
```

**Cap at 25 iterations.** 10 is too limiting for realistic refactoring tasks
(read file → parse → identify changes → edit N files → run tests → fix errors).

**The loop is invisible to the Client.** This means Claude Code (with
`ANTHROPIC_BASE_URL=http://localhost:8080`) gets tool-augmented local responses
with no modification.

## 7. Security and Confirmation Gate

### What the Bridge blocks immediately (no confirmation)

```python
BLOCKED_PATTERNS = [
    r"rm\s+-rf\s+/",
    r"rm\s+-rf\s+~/",
    r"dd\s+.*of=/dev/",
    r"mkfs\.",
    r">\s*/dev/sd",
    r"chmod\s+777\s+/",
    r"curl.*\|\s*sh",
    r"wget.*\|\s*sh",
]

BLOCKED_PATHS = [
    "~/.ssh/",
    "~/.nasim/",
    "~/.claude/",
    "~/.gnupg/",
    "/etc/passwd",
    "/etc/shadow",
]
```

### Confirmation gate protocol

When a `bash` command or `write_file` to an existing file passes the blocklist but
requires confirmation, the Bridge does not pause — it streams a synthetic SSE event:

```
event: confirmation_required
data: {"tool": "bash", "command": "git push --force origin main", "risk": "high"}
```

The Client renders this as an interactive prompt. The user types `y` or `n`. The
Client's response is a new request with the same conversation thread plus a
synthetic `tool_result`:

```json
{"role": "user", "content": [{"type": "tool_result", "tool_use_id": "...", "content": "approved"}]}
```

Or `"denied"` if the user rejected. The Bridge continues the loop from this result.

This keeps the async streaming model clean. No pausing of async loops or shared
mutable state between the confirmation and the tool executor.

### Path scope

Tools operate within the configured workspace root by default. Write access outside
`tools.workspace_root` (default: project working directory) requires explicit path
allowlist entry in `~/.nasim/tools/approved_paths.yaml`.

## 8. Plan Mode

Plan mode is a system-prompt mode, not a special tool.

When the user runs `nasim chat --plan "refactor the auth module"`, the Bridge
prepends a planning system prompt:

```
You are in PLAN MODE. Analyze the task and produce a structured execution plan.
Do NOT call any tools. Output the plan as a numbered list. End with "PLAN COMPLETE".
The user will review and approve before execution begins.
```

The LLM produces a plan without executing anything. The Client renders it and
prompts with `Execute this plan? (y/n/edit)`. On approval, the Bridge re-submits
the original request with execution mode (standard system prompt). The plan is
stored in the session so the LLM has context on what was agreed.

This is simpler than implementing a separate plan mode at the tool executor level
and works with any model that follows instructions.

## 9. Local Knowledge Repository

```
~/.nasim/
├── config.yaml                    # user config
├── CLAUDE.md                      # system prompt injected at every session start
├── history/
│   ├── sessions/
│   │   └── YYYY/MM/
│   │       └── {session_id}.jsonl  # one JSON line per turn
│   └── index.db                   # SQLite FTS5 — built on session close
├── projects/
│   └── {project_hash}/
│       └── CLAUDE.md              # project-scoped system prompt
├── tools/
│   └── approved_paths.yaml        # write-access allowlist
└── tmp/
    └── active_session             # current session ID (for resume)
```

### Session format (JSONL)

One JSON object per line, appended at each turn:

```json
{"turn": 1, "ts": "2026-06-12T10:00:00Z", "role": "user", "content": "refactor auth module", "session_id": "abc123"}
{"turn": 2, "ts": "2026-06-12T10:00:03Z", "role": "assistant", "content": "...", "model": "qwen2.5-coder:32b", "tokens": {"input": 512, "output": 847}, "session_id": "abc123"}
```

JSONL gives append-only writes and crash recovery without WAL configuration.
SQLite is only used for the FTS5 search index, built at session close:

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS turns_fts USING fts5(
    session_id, role, content, ts,
    tokenize='porter unicode61'
);
```

### CLAUDE.md as the sync point

**There is no migration engine.** This is the right call.

Both Nasim and Claude Code read `~/.nasim/CLAUDE.md`. If Nasim produces a useful
insight, Salim edits CLAUDE.md directly (or runs `nasim memory add "..."` which
appends to CLAUDE.md). Claude Code's `state.db` is an internal database — writing
to it from outside would be fragile and potentially corrupt Claude Code's session
state. The sync point is the human-readable, version-controlled CLAUDE.md.

Writing an import/export layer that parses Claude Code's internal JSON session
format is technical debt against a moving target. CLAUDE.md is the stable surface.

### Config schema

```yaml
api:
  base_url: http://localhost:8080   # Bridge address (via SSH tunnel)
  key_env: ANTHROPIC_API_KEY        # env var name

bridge:

  model: qwen2.5-coder:32b

```

## 10. Environment Redirection

The Anthropic Python SDK reads `ANTHROPIC_BASE_URL` natively. No patching required.

```python
import anthropic

client = anthropic.Anthropic(
    base_url=os.environ.get("ANTHROPIC_BASE_URL", "https://api.anthropic.com"),
    api_key=os.environ.get("ANTHROPIC_API_KEY", ""),
)
```

When `ANTHROPIC_BASE_URL` is set, all requests go to the Bridge. The Bridge handles
model mapping from Claude names to local names:

```python
MODEL_MAP = {
    "claude-opus-4-8": "qwen2.5-coder:32b",
    "claude-sonnet-4-6": "qwen2.5-coder:32b",
    "claude-haiku-4-5-20251001": "qwen2.5-coder:7b",
}

def resolve_model(requested: str) -> str:
    return (
        os.environ.get("NASIM_MODEL")  # explicit override
        or MODEL_MAP.get(requested)
        or "qwen2.5-coder:32b"         # fallback
    )
```

## 12. Streaming Architecture

The Bridge emits `text/event-stream` (SSE). Each event maps to an Anthropic
streaming delta type. The Anthropic SDK's `messages.stream()` parses this
transparently:

```
event: message_start
data: {"type": "message_start", "message": {"id": "msg_...", "role": "assistant"}}

event: content_block_start
data: {"type": "content_block_start", "index": 0, "content_block": {"type": "text"}}

event: content_block_delta
data: {"type": "content_block_delta", "index": 0, "delta": {"type": "text_delta", "text": "..."}}

event: message_stop
data: {"type": "message_stop"}
```

Tool calls are buffered in the Bridge until the tool call JSON is complete, then
emitted as a `tool_use` content block before the tool executes.

For the confirmation gate, the Bridge emits a custom event type:

```
event: confirmation_required
data: {"tool": "bash", "command": "...", "risk": "medium", "tool_use_id": "..."}
```

The CLI renderer intercepts this event type before passing to the SDK and renders
the confirmation prompt inline.

## 13. Drop-in Compatibility

Because the Bridge exposes a strict `/v1/messages` endpoint, Claude Code itself
routes through it:

```bash
export ANTHROPIC_BASE_URL=http://localhost:8080
export ANTHROPIC_API_KEY=local
claude "refactor the auth module"
```

Claude Code's tools (Read, Write, Edit, Bash, Glob, Grep) continue to execute on
`salim-hp` as normal — Claude Code handles those itself. The Bridge only handles
the LLM inference routing. This is the right division: Claude Code owns its tool
execution; Nasim owns the model routing.

## 14. Phases

Each phase is independently shippable and useful.

### Phase 1 — Foundation

- [ ] FastAPI Bridge on `black` with pass-through to Ollama `/v1/messages`
- [ ] Anthropic SDK-based CLI client on `salim-hp`
- [ ] SSH tunnel alias
- [ ] JSONL session write on every turn
- [ ] CLAUDE.md injection as system prompt

**Done when:** `nasim chat "hello"` routes to local Ollama and streams the response.

### Phase 2 — Agentic Tooling

- [ ] Tool definitions injected into every Bridge request
- [ ] Tool loop in Bridge (read_file, write_file, bash, list_dir, search)
- [ ] Blocklist enforcement
- [ ] Confirmation gate SSE event + Client rendering
- [ ] Tool iteration cap (25)

**Done when:** `nasim chat "add a docstring to every function in src/"` executes
tool calls, reads files, writes edits, without leaving the chat session.

### Phase 3 — Knowledge Repo

- [ ] SQLite FTS5 index built at session close
- [ ] `nasim search "sqlite session"` queries history
- [ ] Project-scoped CLAUDE.md (`projects/{hash}/CLAUDE.md`)
- [ ] `nasim memory add "..."` appends to global CLAUDE.md
- [ ] Session resume from `tmp/active_session`

**Done when:** `nasim search "the auth bug we fixed"` returns the relevant session.

### Phase 4 — Production Polish

- [ ] OpenAI translation layer (for models without native Anthropic endpoint)
- [ ] LM Studio backend support
- [ ] Plan mode (`--plan` flag)
- [ ] Streaming diff renderer in `rich` (live patch preview before write_file)
- [ ] `nasim serve --daemon` as a systemd service on `black`
- [ ] Token usage display and session cost tracking

**Done when:** Nasim is the daily driver. Claude Code is optional, not required.

## 15. Key Design Decisions

**Bridge on `black`, not `salim-hp`.**
The agentic loop is 3-10 LLM round trips per user turn on average. At 1-3s per
Ollama inference on a local GPU, collocating the loop with the model saves
3-30 seconds of latency per turn versus running the loop on `salim-hp` over LAN.

**No migration engine.**
Claude Code's internal session format (`state.db`, JSON sessions) is undocumented
and will change. Writing against it is maintenance debt. CLAUDE.md is the stable,
human-readable sync surface. If a session insight matters, it gets written to
CLAUDE.md manually or via `nasim memory add`. That's it.

**Confirmation gate as SSE event, not a blocking call.**
Pausing an async loop on `asyncio.Event()` while waiting for user input creates
a shared mutable state problem and makes the Bridge stateful between requests.
Streaming a `confirmation_required` event and receiving approval as a follow-up
request keeps both Bridge and Client stateless. The conversation thread is the state.

**JSONL over SQLite for primary session storage.**
SQLite WAL mode is required for concurrent writes. JSONL never needs configuration —
append is atomic at the filesystem level. The FTS index is read-only after build;
it can be rebuilt from the JSONL files at any time, making it a pure cache.

**Ollama native Anthropic endpoint first.**
Building a translation layer before testing the native endpoint is premature
optimization. Ollama's `/v1/messages` endpoint has been production-ready since
early 2026. Test it, measure tool-call fidelity, and only add translation if
there are concrete gaps.
