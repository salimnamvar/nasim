# locllm — Technical Blueprint

A terminal-native AI coding assistant that runs locally as a sibling to
Claude Code, backed by Ollama or LM Studio, speaking the Anthropic Messages
API natively.

---

## 1. System Overview

```
┌─────────────────────────────────────────────────────────────┐
│  Developer Machine                                          │
│                                                             │
│  ┌──────────────┐   Anthropic     ┌──────────────────────┐ │
│  │   CLI Client │─── Messages ───►│  Bridge Server       │ │
│  │  (locllm)    │◄── (streaming) ─│  :8080               │ │
```

| Local | `http://localhost:8080` | `local` (ignored) |
| Cloud | *(unset)* | real Anthropic key |

## 2. Architecture

### 2.1 CLI Client

Thin terminal process. Responsibilities:

- Read `ANTHROPIC_BASE_URL` / `ANTHROPIC_API_KEY` at startup.
- Send `POST /v1/messages` (Anthropic Messages format) to the resolved base URL.
- Stream the response via Server-Sent Events and render it with
  syntax-highlighted, scrollable terminal output.
- Write each turn to the local history store (`~/.locllm/history/`).
- On tool-use blocks in the response, re-submit the tool result in the
  next `messages` turn — identical to how the Anthropic SDK handles it.

The client is intentionally stateless. All persistence lives in `~/.locllm/`.

### 2.2 Bridge Server

A local HTTP proxy that translates between the Anthropic Messages API
(what the client sends) and the OpenAI Chat Completions API (what Ollama
and LM Studio expose). Responsibilities:

- Accept `POST /v1/messages` (Anthropic format).
- Map to `POST /v1/chat/completions` (OpenAI format) and forward to the
  local LLM backend.
- Map the OpenAI response back to an Anthropic `Message` object (including
  `stop_reason`, `usage`, content blocks).
- Execute tool calls locally (filesystem, shell) when the LLM emits a
  `tool_use` content block.
- Stream the translated response back to the client via SSE.

### 2.3 Local LLM Backend

Ollama (`http://localhost:11434`) or LM Studio (`http://localhost:1234`).
Both expose an OpenAI-compatible `/v1/chat/completions` endpoint.
The Bridge targets whichever is configured in `~/.locllm/config.yaml`.

## 3. API Compatibility Layer

### 3.1 Anthropic Messages → OpenAI Chat Completions

The key translation the Bridge performs on every request:

```
Anthropic Request                 OpenAI Request
─────────────────────             ──────────────────────────
model: "claude-3-5-..."     →     model: "qwen2.5-coder:32b"
system: "You are..."        →     messages[0]: {role: system}
messages: [{role, content}] →     messages[1..]: {role, content}
tools: [{name, input_schema}] →   tools: [{type:function, function}]
max_tokens: 8192            →     max_tokens: 8192
stream: true                →     stream: true
```

Content block arrays (`content: [{type:"text", text:"..."}]`) are
flattened to strings for the OpenAI side; the reverse mapping reconstructs
typed content blocks from the OpenAI text output.

### 3.2 OpenAI Response → Anthropic Message

```
OpenAI Response               Anthropic Response
─────────────────             ────────────────────────────
choices[0].message.content →  content: [{type:"text", text}]
choices[0].finish_reason   →  stop_reason: "end_turn" | "tool_use"
usage.prompt_tokens        →  usage.input_tokens
usage.completion_tokens    →  usage.output_tokens
```

Tool call chunks in streaming mode are buffered until complete, then
emitted as a single `tool_use` content block.

## 4. Environment Redirection

### 4.1 Variable Semantics

```bash
# Route to local Bridge
export ANTHROPIC_BASE_URL=http://localhost:8080
export ANTHROPIC_API_KEY=local

# Route to official Anthropic API
unset ANTHROPIC_BASE_URL
export ANTHROPIC_API_KEY=sk-ant-...
```

The Anthropic Python SDK (`anthropic.Anthropic()`) reads both variables
natively — no patching required. When `ANTHROPIC_BASE_URL` is set, the
SDK sends all requests there instead of `api.anthropic.com`.

### 4.2 Shell Alias Pattern

```bash
# ~/.bashrc
alias locllm='ANTHROPIC_BASE_URL=http://localhost:8080 ANTHROPIC_API_KEY=local locllm'
alias cloudllm='unset ANTHROPIC_BASE_URL locllm'
```

### 4.3 Config Fallback Chain

Resolution order (first defined wins):

1. CLI flag: `--base-url http://...`
2. `ANTHROPIC_BASE_URL` environment variable
3. `~/.locllm/config.yaml` → `api.base_url`
4. Default: `https://api.anthropic.com` (official Anthropic)

## 5. Local Knowledge Repository

### 5.1 Directory Layout

```
~/.locllm/
├── config.yaml              # user config (backend, model, editor, etc.)
├── history/
│   ├── sessions/
│   │   └── YYYY/MM/
│   │       └── {session_id}.jsonl   # one JSON line per turn
│   └── index.db             # SQLite — full-text search over history
├── context/
│   ├── system_prompts/
│   │   └── default.md       # injected as system message
│   └── snippets/            # reusable context fragments
├── tools/
│   └── approved_paths.yaml  # allowlist for filesystem tool access
└── tmp/
    └── last_session.txt     # last active session ID
```

### 5.2 Session Store (`history/sessions/`)

Each session is a `.jsonl` file — one JSON object per line (turn). Format:

```jsonc
{
  "turn": 1,
  "ts": "2026-06-12T10:00:00Z",
  "role": "user",
  "content": "...",
  "model": "qwen2.5-coder:32b",
  "tokens": {"input": 512, "output": 0}
}
```

JSONL gives append-only writes, safe crash recovery, and trivial streaming
reads. No locking needed for sequential sessions.

### 5.3 History Index (`history/index.db`)

SQLite with FTS5 for `--search` queries across all sessions:

```sql
CREATE VIRTUAL TABLE turns_fts USING fts5(
    session_id, role, content, tokenize='porter unicode61'
);
```

Indexed lazily on session close, not on each turn.

### 5.4 Config Schema (`config.yaml`)

```yaml
api:
  base_url: http://localhost:8080   # or unset for cloud
  key_env: ANTHROPIC_API_KEY

bridge:
  backend: ollama                   # ollama | lmstudio
  backend_url: http://localhost:11434
  model: qwen2.5-coder:32b

context:
  system_prompt: default            # name of file in context/system_prompts/
  max_history_turns: 20             # turns injected from history

tools:
  enabled: true
  require_confirmation: true        # prompt before shell execution
```

## 6. Tool Use and the Bridge

### 6.1 Tool Definitions (Anthropic Format)

The Bridge injects these tool definitions into every request to the local
LLM (same as Claude Code's built-in tools):

```python
TOOLS = [
    {
        "name": "read_file",
        "description": "Read a file from the local filesystem.",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"]
        }
    },
    {
        "name": "write_file",
        "description": "Write or overwrite a file.",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string"},
                "content": {"type": "string"}
            },
            "required": ["path", "content"]
        }
    },
    {
        "name": "bash",
        "description": "Execute a shell command. Returns stdout and stderr.",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {"type": "string"},
                "timeout": {"type": "integer", "default": 30}
            },
            "required": ["command"]
        }
    },
    {
        "name": "list_dir",
        "description": "List directory contents.",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"]
        }
    }
]
```

### 6.2 Agentic Tool Loop (Bridge Side)

```
Client sends /v1/messages request
         │
         ▼
Bridge forwards to local LLM (OpenAI format)
         │
         ▼
LLM responds with tool_use block?
    ├── No  → translate to Anthropic Message, stream to client
    └── Yes → execute tool locally
                   │
                   ├── read_file / list_dir / write_file → fs ops
                   └── bash → subprocess with timeout + allowlist check
                   │
                   ▼
              Append tool_result to messages
              Re-submit to LLM (loop)
              (cap at 10 tool iterations)
```

The loop runs inside the Bridge, invisible to the client. The client sees
a single streaming response as if the LLM answered directly.

### 6.3 Security Controls

- **Path allowlist:** `tools/approved_paths.yaml` defines which directories
  `write_file` and `bash` may touch. Default: current working directory only.
- **Command blocklist:** `rm -rf /`, `dd`, `mkfs`, and patterns targeting
  `~/.ssh/`, `~/.locllm/` are rejected before execution.
- **Confirmation gate:** when `tools.require_confirmation: true`, the Bridge
  pauses and streams a confirmation prompt to the client before any `bash`
  or `write_file` execution.
- **Timeout:** all subprocess calls time out at `tools.bash_timeout` seconds
  (default 30). Hanging processes are `SIGKILL`ed.

## 7. Technical Stack

### 7.1 Language: Python 3.12+

Rationale:
- The Anthropic SDK is Python-first; no translation layer needed.
- Ollama and LM Studio both have Python clients.
- `rich` + `typer` give Claude Code-quality terminal rendering in ~200 lines.
- Salim's existing stack — no context switch.

### 7.2 Core Dependencies

| Layer | Library | Why |
|-------|---------|-----|
| CLI | `typer` | Declarative arg parsing; generates `--help` automatically |
| Terminal render | `rich` | Markdown rendering, syntax highlight, live panels, spinners |
| Anthropic client | `anthropic` | Native Messages API; reads `ANTHROPIC_BASE_URL` |
| Bridge HTTP | `fastapi` + `uvicorn` | Async, SSE-native, minimal overhead |
| Outbound HTTP | `httpx` | Async streaming to Ollama/LM Studio |
| Config | `pyyaml` | Simple, human-editable config files |
| History DB | `sqlite3` (stdlib) | Zero-dependency, FTS5 search |
| Process exec | `asyncio.subprocess` | Async shell execution with timeout |
| Testing | `pytest` + `respx` | `respx` mocks httpx calls; no live LLM needed |

### 7.3 Async Strategy

Both the Bridge (FastAPI) and the CLI streaming reader are fully async.
The tool execution loop inside the Bridge uses `asyncio.subprocess` so
long-running commands do not block other in-flight requests.

### 7.4 Streaming (SSE)

The Bridge sends `text/event-stream` responses. Each SSE event maps to one
Anthropic streaming delta type:

```
event: content_block_delta
data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"..."}}

event: message_stop
data: {"type":"message_stop"}
```

The CLI client uses `anthropic.Anthropic().messages.stream()` — the SDK
handles SSE parsing and reconstructs the complete `Message` on close.

## 9. Initial Setup Steps

### Step 1 — Install Ollama and pull a model

```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull qwen2.5-coder:32b
```

### Step 2 — Install locllm

```bash
pip install locllm           # or: pip install -e . from repo root
locllm init                  # creates ~/.locllm/ with defaults
```

### Step 3 — Start the Bridge

```bash
locllm serve                 # starts FastAPI bridge on :8080
# or as a background service:
locllm serve --daemon
```

### Step 4 — Configure environment

```bash
export ANTHROPIC_BASE_URL=http://localhost:8080
export ANTHROPIC_API_KEY=local
```

### Step 5 — Chat

```bash
locllm chat                  # interactive REPL
locllm chat "explain this file" < src/locllm/bridge/translator.py
locllm chat --search "sqlite session"   # search history
```

### Step 6 — Drop-in replacement for Claude Code (optional)

Because the Bridge speaks the Anthropic Messages API, any tool that
respects `ANTHROPIC_BASE_URL` routes to the local LLM automatically —
including Claude Code itself:

```bash
export ANTHROPIC_BASE_URL=http://localhost:8080
claude "refactor this function"   # now hits local LLM
```

## 10. Phase Boundaries

| Phase | Scope |
|-------|-------|
| P1 — Core | Bridge server, format translation, Ollama backend, CLI chat loop, JSONL history |
| P2 — Tools | Tool definitions, agentic loop, bash executor, confirmation gate |
| P3 — Context | FTS history index, system prompt injection, snippet manager |
| P4 — Parity | LM Studio backend, streaming diff renderer, multi-session workspace |

P1 alone makes the app useful. Each phase is independently shippable.

## 11. Key Design Decisions

**Bridge vs pure client-side translation.** A server-side Bridge (rather
than a client-side shim) keeps the CLI simple, allows multiple clients
(Claude Code, Cursor, raw curl) to share one local LLM session, and lets
tool execution happen in a controlled process with proper timeouts.

**JSONL over SQLite for session storage.** Append-only writes survive
crashes without WAL configuration. SQLite is used only for the FTS index
built at session close — not as the primary store.

**Tool loop in the Bridge, not the client.** Hiding the agentic loop from
the client means existing clients (Claude Code via `ANTHROPIC_BASE_URL`)
get tool-augmented responses without any modification.

**No streaming LLM state across restarts.** The Bridge is stateless between
requests. The local LLM's context window is reconstructed from history on
each request. This keeps the Bridge horizontally restartable and avoids
context drift.
