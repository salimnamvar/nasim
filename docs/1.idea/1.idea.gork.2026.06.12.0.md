**Technical Blueprint: Local Claude Code Sibling (e.g., "Lumina Code" or "Forge CLI")**

This blueprint outlines building a terminal-native AI coding assistant that mirrors Claude Code's UX and capabilities while running fully locally or in a hybrid setup. It follows a **Client-Server architecture**, prioritizes **native Anthropic Messages API compatibility**, supports environment-based redirection, includes a robust local knowledge repo, and features a tool-use bridge for agentic capabilities like file edits and shell commands.

### 1. High-Level Architecture (Client-Server Model)

- **Client (CLI on salim-hp or any dev machine)**: Interactive TUI/CLI that handles user input, conversation history, tool orchestration (with human approval), real-time terminal output, and Git integration. It sends requests in **Anthropic Messages format** (`/v1/messages`).
- **Server (on black or local)**: Hosts the LLM (via Ollama, LM Studio, vLLM, etc.) and exposes an **Anthropic-compatible endpoint**. Optionally includes a lightweight proxy/bridge for tool use.
- **Communication**: HTTP/REST (or WebSocket for streaming). Client uses env vars to point to local or remote server.
- **Fallback**: Seamless toggle between local LLM and official Anthropic via env vars.

**Core Flow**:
1. User runs `forge` (or your CLI name) in a project dir.
2. Client loads local context/repo state.
3. Sends Anthropic-formatted messages to server.
4. Server (or proxy) translates to LLM backend if needed, executes tools if requested, returns response.
5. Client applies edits, runs commands (with approval), updates local state.

### 2. Technical Stack Recommendations

Based on tools like **Aider** (Python, Git-native, tree-sitter for code understanding), **OpenCode** (Go-based TUI/CLI), and **Cline** (TypeScript/JS with strong tool/agent support):

- **Language**: **Python** (recommended for rapid prototyping, rich ecosystem) or **Go** (for performance, lightweight binaries like OpenCode).
  - Python: Excellent for LLM integrations, shell execution (`subprocess`, `asyncio`), TUI (`Textual` or `rich`), and Git (`dulwich` or `GitPython`).
  - Go: Faster startup, great for cross-platform CLI/TUI (`bubbletea`/`lipgloss`), secure exec.
- **Key Libraries**:
  - **API/HTTP**: `httpx` (Python) or `net/http` + `chi` (Go) for Anthropic-compatible server. `anthropic` SDK for client-side.
  - **TUI/Real-time Output**: Python `Textual` (rich interactive UI with streaming) or Go `bubbletea`. Use `asyncio`/`goroutines` for non-blocking terminal I/O and streaming responses.
  - **Shell Execution**: Python `asyncio.subprocess` or Go `os/exec` with sandboxing (e.g., run in project dir only, timeout, approval prompts). Use `pty` for interactive commands.
  - **Code Understanding**: `tree-sitter` (via Python bindings or Go) for AST parsing across languages, like Aider.
  - **Local LLM**: Integrate with Ollama Python/Go clients or OpenAI/Anthropic compatible clients.
  - **Other**: `pydantic` for schema validation, `sqlite` + `sqlmodel` (or embedded KV like `bbolt` in Go) for state, `diff-match-patch` for safe edits.
- **Why this stack?** Balances developer experience (Python's ecosystem) with performance/security (Go). Aider proves Python excels for Git-aware coding agents; OpenCode shows Go's strength for terminal tools.

Start with Python for MVP, then consider Go rewrite for distribution.

### 3. Initial Setup Steps

**Project Structure** (monorepo or separate client/server):
```
forge-cli/
├── client/          # CLI/TUI
├── server/          # Proxy + Anthropic compat layer
├── bridge/          # Tool execution logic
├── local-repo/      # Default ~/.forge/
├── pyproject.toml / go.mod
└── docs/
```

**Setup Steps**:
1. **Initialize Repo**: `git init`, add `.gitignore` (ignore large models, temp files).
2. **Dependencies**: Install core libs (e.g., `pip install httpx textual pydantic anthropic ollama`).
3. **Server Setup**:
   - Run Ollama/LM Studio/vLLM on `black`.
   - Implement or use a lightweight Anthropic proxy (e.g., inspired by LiteLLM or claude-code-proxy).
4. **Client Install**: `pip install -e .` or build binary. Support `forge --model qwen3-coder`.
5. **Env-Based Redirection** (core feature):
   ```bash
   # In ~/.bashrc or session
   export FORGE_BASE_URL="http://<BLACK_IP>:11434"  # or http://localhost:4000 for proxy
   export FORGE_API_KEY="local"  # Placeholder; required for SDK compatibility
   export FORGE_DEFAULT_MODEL="qwen3-coder"  # Maps Claude names internally
   ```
   - Client code: Check `os.getenv("FORGE_BASE_URL")`; if set and not Anthropic default, use it. Fallback to `https://api.anthropic.com`.
   - Use `anthropic.Anthropic(base_url=..., api_key=...)` for native client.

### 4. API Integration (Anthropic Messages Compatibility)

- **Goal**: Client always speaks native Anthropic `/v1/messages` (with tools, streaming, vision).
- **Server/Proxy Implementation**:
  - Expose `/v1/messages` endpoint that accepts Anthropic schema.
  - If backend is Ollama/OpenAI-compatible: Translate (map `messages`, `tools` to chat completions; handle tool calls in response).
  - Use libraries like LiteLLM for heavy lifting, or implement a thin proxy in Python/Go.
  - Support model mapping: `claude-3.5-sonnet` → your local model via env var or config.
  - Streaming: Use Server-Sent Events (SSE) for real-time tokens in TUI.
- **Client**:
  - Use official Anthropic SDK with custom base_url.
  - Handle `/models` endpoint for discovery (query server on startup).
  - Implement conversation turns with system prompts for coding persona ("You are a terminal-native coding assistant...").

Test: `curl` Anthropic requests against your server; verify with Claude Code env vars pointing to it.

### 5. Local Knowledge Repo & State Management

Mirror `~/.claude/` for persistence and performance.

**Structure** (`~/.forge/` or project-specific `.forge/`):
```
~/.forge/
├── config.yaml          # Models, providers, rules
├── conversations/       # JSONL or SQLite per session (id, messages, metadata)
├── history/             # Global chat logs (timestamped)
├── knowledge/           # User-provided docs, embeddings (optional vector DB)
├── cache/               # Code embeddings, file index (gitignored large files)
├── projects/            # Per-project indexes (file tree, last-modified)
└── state.db             # SQLite: sessions, checkpoints, tool history
```

**Performance Optimizations**:
- **Indexing**: On project load, build lightweight index with `tree-sitter` + file metadata. Use `git ls-files` for tracked files.
- **Database**: SQLite with indexes on timestamps/conversation IDs. For large repos, embed optional vector search (ChromaDB or LanceDB) for RAG over codebase.
- **Context Management**: Summarize old messages; use "map" of codebase (like Aider) to selectively include files.
- **Git Integration**: Auto-commit changes with meaningful messages (user approval). Support `/undo`, checkpoints.
- **Security**: Store nothing sensitive; respect `.gitignore`; run tools in project cwd with timeouts.

Load context on startup: Scan cwd, read relevant files based on user query.

### 6. Tool Use & The Bridge (Agentic Capabilities)

**Plan**:
- **Tool Definitions**: Define Anthropic-style tools in client (e.g., `read_file`, `edit_file`, `run_shell`, `search_code`, `git_commit`).
- **Bridge/Proxy Layer**:
  - Run as separate process or integrated server module on `black`.
  - When LLM requests tool use, client intercepts, shows preview/approval UI, executes safely, feeds result back in next message.
  - For advanced execution: Sandboxed shell (e.g., restricted `bash` with `timeout`, no network unless allowed).
  - Translation: If LLM backend doesn't support tools natively, proxy simulates them (e.g., convert to function calling).
- **Implementation**:
  - Python: `subprocess` + approval prompt in TUI.
  - Secure: Whitelist commands, diff previews for edits (`unified_diff`), read-only for sensitive paths.
  - Inspired by Cline's human-in-the-loop and claude-proxy.

**Example Tool Flow**:
1. LLM outputs tool call.
2. Client: `Apply edit to main.py? [y/n/diff]`
3. Execute → Append result to conversation → Continue.

### 7. Development & Next Steps

1. **MVP**: Basic CLI chat with local Ollama + env redirection + simple file read/write.
2. **Iterate**: Add TUI, Git, full tools, model mapping.
3. **Distribution**: PyPI / single binary (PyInstaller or Go build). Cross-platform.
4. **Testing**: Unit tests for API compat, integration with real local models, safety (e.g., no destructive commands without approval).
5. **Advanced**: RAG over repo, multi-agent (architect/editor like Aider), LSP integration.
